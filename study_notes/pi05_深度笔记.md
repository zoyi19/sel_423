### 1.3 在 VLA 谱系中的位置

主流 VLA 五条技术路线:

| 路线 | 代表模型 | Action 表示 | VLM backbone |
|---|---|---|---|
| **离散 token AR** | OpenVLA, OpenVLA-OFT | 离散 token(自回归生成) | LLaMA-2-7B / Prismatic |
| **Flow Matching** | **π0 / π0.5** | 连续 action via flow matching | PaliGemma-3B |
| **Diffusion** | RDT-1B, Diffusion Policy | 连续 action via DDPM | T5-XXL / 自定义 |
| **DiT(diffusion transformer)** | GR00T N1.5 / N1.7 | 连续 action via DiT | Eagle-2.5 / Cosmos-Reason |
| **离散频域 token** | π0-fast | FAST tokenization | PaliGemma-3B |

**π0.5 的差异化**:
- 比 OpenVLA **action 更平滑**(连续 vs 离散)
- 比 Diffusion 系 **推理更快**(10 步 ODE vs 50-100 步 DDPM)
- 比 Diffusion Policy **更"模型驱动"**(VLM 主导而不是仅 vision)
- 比 GR00T **VLM 更小**(3B vs Eagle 2.5 ~3B 但 Cosmos 8B+)

### 1.4 为什么 π0.5 重要

1. **首次系统证明** VLM-based VLA 能在**完全陌生的真实家庭**做长视地任务(扫地 / 整理床铺)
2. **架构稳定**:之后 π*0.6 / π0.7 都基于 π0.5 主架构,只改训练方法
3. **开源**:openpi 仓库公开,**学术 + 工业都能复现**(对比 OpenVLA-OFT 闭源 fine-tune 配方)
4. **实习直接相关**:你的项目对接过 π0.5,简历能讲深度

### 2.2 三个核心组件

#### (a) VLM Backbone:**PaliGemma-3B**

`PaliGemma-3B = SigLIP-So400m/14 (vision) + Gemma-2.6B (LLM)`

| 组件 | 作用 | 参数量 |
|---|---|---|
| SigLIP-So400m/14 | 把图像 → vision tokens(每张图 ~256 tokens) | 0.4B |
| Gemma-2.6B | 处理 [vision tokens, text tokens] | 2.6B |

**为什么选 PaliGemma 不是 LLaMA / Qwen**:
1. **PaliGemma 是 Google 专门为 VLM 训练的**(图文对齐已经做好),省掉自己融合视觉的工作
2. **Gemma 比 LLaMA 小**(2.6B vs 7B),真机推理更现实
3. **SigLIP 比 CLIP 在小尺寸下表现更稳**(no temperature trick)
4. **BSD-3 license**,商用友好(LLaMA-2 的 license 限制更多)

#### (b) Action Expert:小型 Gemma 变体

不是单独 transformer,而是 **共享 PaliGemma 的 attention 计算**(prefix-LM 结构)。具体看代码 §五。

#### (c) Flow Matching Head

最后一层 `Linear(width → action_dim)` 投影,把 action expert 的 token 映射回 action space。**速度场 (velocity field) 预测**而不是直接预测 action。

### 2.3 输入输出 Schema

**输入**:
```python
Observation = {
    "images": {                  # multi-camera dict
        "head": (B, 3, 224, 224),
        "left_wrist": (B, 3, 224, 224),
        "right_wrist": (B, 3, 224, 224),
    },
    "image_masks": {...},         # 哪些相机本帧有效

    "tokenized_prompt_mask": (B, 48),

```

**输出**:


## 三、训练策略

### 3.1 数据组成

π0.5 论文【来源:[arxiv 2504.16054](https://arxiv.org/abs/2504.16054)】明确提到 **"hybrid multi-modal examples"**:

| 数据类型 | 用途 | 规模(估计) |
|---|---|---|
| Real-robot data | 主要 IL 训练 | 100M+ steps,跨 70+ 机器人 |
| OXE(Open X-Embodiment) | cross-embodiment 基础 | ~1M episodes |
| Web image-text | 保留 PaliGemma 的 grounding | OWLv2 / WebLI 等 |
| Object detections | 中间监督信号 | YOLO / OWLv2 标注 |
| Semantic subtask labels | 高层次 reasoning | 人工标注或 LLM 标注 |

### 3.2 Co-training 的具体做法

不是简单 concat 数据集,而是 **同一个 forward pass 里混合不同模态的例子**。比如同一 batch 里:
- 30% 是 (image, text → action) IL 任务
- 30% 是 (image, text → bounding box) detection 任务
- 20% 是 (image → text) captioning 任务
- 20% 是 (image, text → semantic subtask) 高层规划任务

**为什么这样做?**
- 防止 VLM 在 IL 训练中**遗忘 grounding 能力**(catastrophic forgetting)
- detection / subtask 等任务**强迫模型保持视觉-语言对齐**
- 推理时 VLA 能力更强:看到陌生物体不会"短路"

### 3.3 Fine-tune 路径

下游任务 fine-tune 通常的做法:
1. **Freeze VLM**(SigLIP 一定 freeze,Gemma 可选 freeze)
2. **Train action expert + flow matching head**
3. **少量真机数据**(几百 episode 即可)

## 四、关键技术点

### 4.1 Flow Matching action head(vs Diffusion)

**Flow Matching** 是 **Rectified Flow / Conditional Flow Matching**(2022-2023)的工程化:

**核心想法**(数学最少版):
- 学一个**速度场** v(x_t, t):告诉你"在 t 时刻,如果你在 x_t,应该往哪个方向走"
- 训练:从 x_0 = noise, x_1 = action 出发,中间点 x_t = t*noise + (1-t)*action,真实速度 u_t = noise - action
- Loss:`L = ||v(x_t, t) - u_t||²`(MSE)
- 推理:从 noise 出发,跑 ODE `dx/dt = v(x, t)`,数值积分到 action

**vs Diffusion (DDPM)**:

| | Flow Matching | Diffusion (DDPM) |
|---|---|---|
| 数学路径 | 直线(x_t = t·noise + (1-t)·action) | 曲线(SDE) |
| 训练 loss | 速度场 MSE | 噪声预测 MSE |
| 推理步数 | 10 步够 | 50-100 步 |
| 收敛速度 | 快 | 慢 |
| 工程稳定性 | 高 | 中 |

**为什么 flow matching 更好(对真机)**:
- **直线轨迹**比扩散过程更短,**10 步 ODE 就够**(对应 100Hz 控制下推理仍然实时)
- **训练 loss 形式简单**(直接 MSE 在速度场上),收敛快

### 4.2 PaliGemma backbone(为什么不是 LLaMA / Qwen)

**已在 §2.2 详述**。补充一点:
- **跨语言不重要**(机器人指令通常英文短句),Gemma 英文足够
- **小尺寸是为了真机推理**(Orin / 4090 都跑得动)

### 4.3 π0 vs π0.5 的架构差异(从代码读出)

【来源:`/home/leju/selbst/openpi/src/openpi/models/pi0.py`】

代码里有一个 `config.pi05` 布尔标志,对应三处差异:

```python
# pi0.py 第 80 行
llm.lazy_init(rngs=rngs, method="init",
              use_adarms=[False, True] if config.pi05 else [False, False])
```

**π0.5 三处变化**:

1. **action expert 用 adaRMS**(adaptive RMSNorm,类似 DiT 的 adaLN)
   - π0:RMSNorm 是固定参数
   - π0.5:RMSNorm 的 scale 由 time embedding 动态生成(类似 DiT 的设计)

2. **去掉单独的 state token**
   ```python
   # pi0.py 第 151-157 行
   if not self.pi05:
       # add a single state token
       state_token = self.state_proj(obs.state)[:, None, :]
       tokens.append(state_token)
       ...
   ```
   - π0:state 作为单独一个 token 加入 attention(占 1 个位置)
   - π0.5:**state 不再单独嵌入**(可能融合到 image / language 的 prompt 里,或 PI 团队认为 vision 已经包含 state 信息)

3. **time embedding 处理方式**
   ```python
   # pi0.py 第 162-178 行
   if self.pi05:
       # time MLP (for adaRMS)
       time_emb = self.time_mlp_in(time_emb)
       time_emb = nnx.swish(time_emb)
       time_emb = self.time_mlp_out(time_emb)
       time_emb = nnx.swish(time_emb)
       action_expert_tokens = action_tokens
       adarms_cond = time_emb
   else:
       # mix timestep + action information using an MLP (no adaRMS)
       ime_tokens = einops.repeat(time_emb, "b emb -> b s emb", s=self.action_horizon)
       action_time_tokens = jnp.concatenate([action_tokens, time_tokens], axis=-1)
       ...
   ```
   - π0:time emb **拼接到 action token 上**,过 MLP
   - π0.5:time emb 单独过 MLP,**作为 adaRMS 的条件**(更优雅)

### 4.4 Cross-embodiment Co-training

70+ 机器人混训的关键 trick:
- **Action 标准化**:不同机器人 joint 数不同,但都 padding 到一个固定长度(比如 32 维),无效维度 mask 掉
- **State padding 同样处理**
- **Embodiment ID 不显式输入**,模型自己从 visual context 推断

> 这也是为什么用户实习时,**Lingbot-VLA 用 1000 条无本体 UMI 数据训只到 30%** —— UMI 本质是新 embodiment,没有大量 cross-embodiment 数据兜底,泛化崩溃。

```python
        )

        if config.pi05:

        else:
            # π0: state 单独投影 + action+time concat 后 MLP

            self.action_time_mlp_in = nnx.Linear(2 * action_expert_config.width, action_expert_config.width, rngs=rngs)
            self.action_time_mlp_out = nnx.Linear(action_expert_config.width, action_expert_config.width, rngs=rngs)
```

#### 片段 2:Flow Matching 训练 loss

【`src/openpi/models/pi0.py:188-214`】

```python
def compute_loss(self, rng, observation, actions, *, train=False):
    preprocess_rng, noise_rng, time_rng = jax.random.split(rng, 3)
    observation = _model.preprocess_observation(preprocess_rng, observation, train=train)

    batch_shape = actions.shape[:-2]
    noise = jax.random.normal(noise_rng, actions.shape)

    # 关键 trick:time 不是均匀采样,而是 Beta(1.5, 1) 分布
    # 这让 t 偏向 [0, 1) 的更小值,即更靠近"target action"的时刻
    # 直观:让模型在"接近最终答案"时学得更精细
    time = jax.random.beta(time_rng, 1.5, 1, batch_shape) * 0.999 + 0.001
    time_expanded = time[..., None, None]

    # x_t = t·noise + (1-t)·action(线性插值)
    x_t = time_expanded * noise + (1 - time_expanded) * actions

    # 真实速度场 u_t = noise - action(常数,因为路径是直线)
    u_t = noise - actions

    # 把 prefix(image+text) 和 suffix(state+action+time)一起 forward
    prefix_tokens, prefix_mask, prefix_ar_mask = self.embed_prefix(observation)
    suffix_tokens, suffix_mask, suffix_ar_mask, adarms_cond = self.embed_suffix(observation, x_t, time)
    input_mask = jnp.concatenate([prefix_mask, suffix_mask], axis=1)
    ar_mask = jnp.concatenate([prefix_ar_mask, suffix_ar_mask], axis=0)
    attn_mask = make_attn_mask(input_mask, ar_mask)
    positions = jnp.cumsum(input_mask, axis=1) - 1

    (prefix_out, suffix_out), _ = self.PaliGemma.llm(
        [prefix_tokens, suffix_tokens], mask=attn_mask, positions=positions,
        adarms_cond=[None, adarms_cond],                  # π0.5 才有 adarms_cond
    )
    # 预测速度场
    v_t = self.action_out_proj(suffix_out[:, -self.action_horizon :])

    # MSE loss on velocity field
    return jnp.mean(jnp.square(v_t - u_t), axis=-1)
```

#### 片段 3:推理(Euler ODE + KV cache)

【`src/openpi/models/pi0.py:216-279`】

```python
def sample_actions(self, rng, observation, *, num_steps=10, noise=None):
    observation = _model.preprocess_observation(None, observation, train=False)
    # 注意:t=1 是 noise,t=0 是 target(和 π0 论文相反,作者承认了)
    dt = -1.0 / num_steps                              # 每步前进 -dt
    batch_size = observation.state.shape[0]
    if noise is None:
        noise = jax.random.normal(rng, (batch_size, self.action_horizon, self.action_dim))

    # === 关键加速:prefix 只 forward 一次,缓存 KV ===
    prefix_tokens, prefix_mask, prefix_ar_mask = self.embed_prefix(observation)
    prefix_attn_mask = make_attn_mask(prefix_mask, prefix_ar_mask)
    positions = jnp.cumsum(prefix_mask, axis=1) - 1
    _, kv_cache = self.PaliGemma.llm([prefix_tokens, None], mask=prefix_attn_mask, positions=positions)

    # === Euler ODE 步进:每步只 forward suffix(action+time),用缓存的 prefix KV ===
    def step(carry):
        x_t, time = carry
        suffix_tokens, suffix_mask, suffix_ar_mask, adarms_cond = self.embed_suffix(
            observation, x_t, jnp.broadcast_to(time, batch_size)
        )
        suffix_attn_mask = make_attn_mask(suffix_mask, suffix_ar_mask)
        prefix_attn_mask = einops.repeat(prefix_mask, "b p -> b s p", s=suffix_tokens.shape[1])
        full_attn_mask = jnp.concatenate([prefix_attn_mask, suffix_attn_mask], axis=-1)
        positions = jnp.sum(prefix_mask, axis=-1)[:, None] + jnp.cumsum(suffix_mask, axis=-1) - 1

        (prefix_out, suffix_out), _ = self.PaliGemma.llm(
            [None, suffix_tokens],                         # prefix 不再 forward
            mask=full_attn_mask, positions=positions,
            kv_cache=kv_cache,                             # 复用 prefix KV
            adarms_cond=[None, adarms_cond],
        )
        v_t = self.action_out_proj(suffix_out[:, -self.action_horizon :])
        return x_t + dt * v_t, time + dt                  # Euler step

    def cond(carry):
        x_t, time = carry
        return time >= -dt / 2

    # JAX while_loop:静态展开 num_steps 次
    x_0, _ = jax.lax.while_loop(cond, step, (noise, 1.0))
    return x_0
```

**关键工程亮点**:
- **prefix KV cache**:VLM 部分(image + language)只 forward 一次,**节省 90%+ 计算量**(因为 ODE 跑 10 步,每步如果都重算 prefix 就太亏)
- **JAX while_loop**:JIT 编译时把循环展开,GPU 利用率高
- **dt 方向反过来**:`dt = -1/num_steps`,从 t=1 走到 t=0(代码注释里作者道歉了)

#### 片段 4:Prefix-LM Attention Mask

【`src/openpi/models/pi0.py:19-44`】

```python
def make_attn_mask(input_mask, mask_ar):
    """
    构造混合注意力 mask:
    - mask_ar=False 的 token 可以互相 attend(prefix 部分:image+text)
    - mask_ar=True 是因果边界
    
    举例:
      [1 1 1 1 1 1] → 纯因果(causal)
      [0 0 0 1 1 1] → prefix-LM(前 3 个互相看,后 3 个 causal)
    """
    mask_ar = jnp.broadcast_to(mask_ar, input_mask.shape)
    cumsum = jnp.cumsum(mask_ar, axis=1)                  # 累计和
    # token i 能 attend token j 当且仅当 cumsum[i] >= cumsum[j]
    attn_mask = cumsum[:, None, :] <= cumsum[:, :, None]
    valid_mask = input_mask[:, None, :] * input_mask[:, :, None]
    return jnp.logical_and(attn_mask, valid_mask)
```

> 这是 **prefix-LM 设计**:image + text 之间双向 attend(不是 causal),action tokens 对前面是 causal。这种设计让 VLM 输出更稳定。

#### 片段 5:Embed Prefix(image + language)

【`src/openpi/models/pi0.py:105-137`】

```python
def embed_prefix(self, obs):
    """把 multi-camera images + tokenized prompt 编码成 prefix tokens"""
    input_mask, ar_mask, tokens = [], [], []

    # === 1. 视觉部分:每个相机 → SigLIP → image tokens ===
    for name in obs.images:
        image_tokens, _ = self.PaliGemma.img(obs.images[name], train=False)
        tokens.append(image_tokens)
        input_mask.append(einops.repeat(obs.image_masks[name], "b -> b s", s=image_tokens.shape[1]))
        ar_mask += [False] * image_tokens.shape[1]            # image tokens 互相 attend

    # === 2. 语言部分:prompt → embed ===
    if obs.tokenized_prompt is not None:
        tokenized_inputs = self.PaliGemma.llm(obs.tokenized_prompt, method="embed")
        tokens.append(tokenized_inputs)
        input_mask.append(obs.tokenized_prompt_mask)
        ar_mask += [False] * tokenized_inputs.shape[1]        # text tokens 也是 prefix(双向)

    return jnp.concatenate(tokens, axis=1), jnp.concatenate(input_mask, axis=1), jnp.array(ar_mask)
```

## 六、工程小巧思

### 6.1 Beta(1.5, 1) 时间分布抽样

```python
time = jax.random.beta(time_rng, 1.5, 1, batch_shape) * 0.999 + 0.001
```

**为什么不是 uniform**?
- Beta(1.5, 1) 的密度在 [0, 1] 上**偏向 0**(也就是偏向"接近 target action"的时刻)
- 直观:让模型在"接近最终答案"时学得更精细,因为这阶段的速度场最重要(直接决定输出)
- 这是 PI 团队实验试出来的,不是数学推导

### 6.2 chunk_size 50 + ODE 10 步

| 参数 | 值 | 意义 |
|---|---|---|
| `action_horizon` | 50 | chunk 长度,控制 100Hz 下覆盖 0.5s |
| `num_steps`(ODE) | 10 | Euler 数值积分步数 |
| 推理时间(H100) | ~50ms / chunk | 每秒可以推 20 个 chunk(20Hz 决策) |

### 6.3 KV cache:加速 90%

如 §5.2 片段 3 所示,prefix(image+language token)只 forward 一次,suffix 每 ODE 步迭代。
- 不用 KV cache:每步 forward 整个 transformer = 10x 计算
- 用 KV cache:每步只 forward suffix(50 个 action tokens),节省 90%+

### 6.4 LeRobot v2 数据格式

openpi **完全对接 LeRobot v2**:
- parquet 存 state / action
- mp4 存 video
- jsonl 存 episode 索引

> 这个对你**实习写的 `bag2lerobot.py` 转换器**完全兼容 —— 你的数据可以直接喂给 openpi fine-tune。

## 八、进化路线

### 8.1【已发布】π0 → π0-fast → π0.5 → π*0.6 → π0.7

#### π*0.6:RECAP RL-from-experience(2025.11.17,arxiv 2511.14759)

**Source**:[π*0.6 a VLA That Learns From Experience](https://arxiv.org/abs/2511.14759), [PI 官博](https://www.pi.website/blog/pistar06), [Humanoids Daily 报道](https://www.humanoidsdaily.com/news/physical-intelligence-claims-rl-is-back-with-new-model-that-learns-from-its-own-mistakes)

**核心创新**:**RECAP**(RL with Experience and Corrections via Advantage-conditioned Policies)

三阶段训练:
1. **Offline RL pretrain**:在大规模历史数据上预训练(advantage-conditioned)
2. **Demos fine-tune**:在 task-specific demos 上做 IL fine-tune
3. **On-robot RL**:真机部署,**收集失败 + 专家干预**,RL 微调

**关键 trick:Advantage-conditioned policy**
- 不是输出"动作",而是输出"在给定 advantage 条件下的动作"
- 训练时 condition on 真实 advantage
- 推理时永远 **condition on 'good'**(高 advantage 标签)→ 自然倾向高 reward 行为

**实测战绩**【来源:PI 官博】:
- **2x+ throughput on hard tasks**(叠衣 / 咖啡机)
- **~50% fewer failures**
- **13 小时连续部署**

**面试可讲**:π*0.6 把"真机 RL 不现实"这条业界共识推翻了一半 —— 不是不能做,是**之前没这么聪明的训练范式**。

#### π0.7:Compositional Generalization(2026.Q1)

**Source**:[Humanoids Daily 报道](https://www.humanoidsdaily.com/news/physical-intelligence-unveils-0-7-the-rise-of-compositional-generalization-in-robotics)

**核心改进**(已公布):
- **任务组合泛化**:见过 "拿杯子" + "倒咖啡",可以组合做 "拿杯子并倒咖啡"
- **多步规划增强**:Long-horizon 任务上更稳

### 8.2【社区推测】(可能的下一步)

- **π1.0**(假想):整合 RL + 任务组合 + reasoning,接近 "general purpose" 通用机器人模型
- **π0.5-mini**:小尺寸蒸馏版本,边端部署(Jetson Nano 级别)
- **π + World Model**:与 V-JEPA 类世界模型集成,让 VLA 有"想象未来"的能力

### 8.3【我的判断】

基于 π 系列发布节奏(2 个月一版),2026 年下半年大概率会出 **π1.0**,主打**整合 RL(从 π*0.6)+ 组合泛化(从 π0.7)+ 可能的世界模型**。

PI 团队的方向论:**"基础模型应该是物理的"** —— 不会做 LLM 的事(reasoning),但会**坚持做**:数据规模化、真机闭环、跨机型泛化。

## 九、面试速答

### 9.1 30 秒版

> "π0.5 是 Physical Intelligence 2025 年发的 VLA 模型,**架构 = PaliGemma-3B + Flow Matching action head**,核心创新是 **跨任务 / 跨机器人的 heterogeneous co-training**,让一个模型能在陌生家庭做扫地、整理床这种长视地家务任务。"

### 9.2 2 分钟版

> 30 秒版 +
>
> "技术上有三个亮点。
>
> 一,**Flow Matching 比 Diffusion 推理更快** —— 直线轨迹 + 10 步 ODE,vs DDPM 的 50-100 步。
>
> 二,**Co-training 设计**:同一 batch 里混合 IL action / detection / captioning / subtask 多模态任务,防止 VLM 在 IL 训练中遗忘 grounding 能力。
>
> 三,**架构上 π0.5 vs π0 的差异**:加了 adaRMS time conditioning(类似 DiT)、去掉了单独的 state token、time embedding 单独 MLP。
>
> 我实习时对接过 π0.5 fine-tune 流程,在 H100 跑 3K step,但因为我们的数据规模和 PI 论文里相比小 100 倍,效果没做出来,后来切回 GR00T 路线。"

### 9.3 5 分钟版(讲透架构 + 进化)

> 2 分钟版 +
>
> "**最有意思的是 π 系列的进化逻辑**。
>
> π0(2024.10)是基础版,首次系统证明 VLM + Flow Matching 可用。
>
> π0-fast(2024.12)是工程优化版,用 FAST tokenization 把推理变快但精度有损。
>
> π0.5(2025.04)是泛化突破版,heterogeneous co-training。
>
> **π*0.6(2025.11)是范式突破** —— 提出 RECAP(RL with Experience and Corrections via Advantage-conditioned Policies)三阶段训练:offline RL pretrain → demos fine-tune → on-robot RL with corrections。**实测 hard tasks 有 2x+ throughput,~50% 更少失败,13 小时连续部署**。
>
> RECAP 的关键 trick 是 **advantage-conditioned policy** —— 训练时 condition on 真实 advantage,推理时永远 condition on 'good',让模型自然倾向高 reward 行为。这个设计借鉴了 LLM 的 advantage-conditioned 思路(类似 Decision Transformer),很优雅。
>
> π0.7(2026.Q1)主打 compositional generalization,见过原子任务能组合做复合任务。
>
> 我的判断是 **π1.0 大概率 2026 年下半年出来**,整合 RL + 组合泛化,可能加上世界模型集成。"
