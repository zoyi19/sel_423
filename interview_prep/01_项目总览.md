## 三、业务背景

- **目标场景**:人形机器人在仓储 / 工业场景完成"**搬箱**(从 A 位置搬到 B 位置)"和"**拆垛**(从多层堆叠中依次取出)"两类任务
- **硬件**:Kuavo 人形机器人(双臂构型 + 轮臂构型 MT-9/MT-7),4 摄视觉(头 + 胸部低视角 + 双肩 realsense/奥比中光),ZED-M 双目 + FoundationStereo 深度估计
- **业务挑战**:
  1. **长视地任务**(视觉伺服→走路→弯腰→抓取→起身→搬运→放下,多个动作串联)
  2. **多构型泛化**(4311 / 4322 / 4611 / 4633 等箱体尺寸差异大)
  3. **实时性要求**(Orin 嵌入式平台,需要 30Hz 推理)
  4. **OOD 泛化**(训练场景是白底,部署要面对杂乱背景、金属重物、倾斜摆放)

## 五、主项目架构 —— 1 主 + 3 子模块

### 模块 1:数据工程 & 增强管线 —— **【个人主力】**

**为什么重要**:VLA 模型成功率瓶颈最终大多落在**数据**上。训练集单调 → OOD 就崩(Groot 在倾斜摆放成功率 **0%** 就是最好证明)。我的工作重点是**让数据分布尽可能靠近部署场景**。

**我主导的工作**:
1. **Cosmos-transfer 背景/光线合成**(2025.12)
   - 基于 NVIDIA Cosmos-transfer 的可控视频生成能力,给真机采集的搬箱/拆垛数据做背景替换和光线变换
   - 目标:把 OOD 杂乱背景成功率从 83% 往 90%+ 推,把倾斜摆放从 0% 拉起来
2. **SAM3+IC-light 备选方案**(2025.12)
   - Cosmos-transfer 不稳定时的 fallback:SAM3 分割前景 → IC-light 重新光照渲染
   - 工程上更轻量、更可控,可与 Cosmos-transfer 组合使用

**我参与并熟悉的工作**:
3. **DINO+SAM+LaMa 自动擦 TAG 流水线**(2025.10,周贤涵主导)
   - 痛点:真机采集时箱子贴了 AprilTag 做视觉伺服,模型会把 TAG 当强特征过拟合
   - 流水线:DINO 检测 TAG 框 → SAM 生成掩码 → LaMa 修复图像
   - 性能:单相机 parquet 8min,4/5 层 4 相机全数据集 **6 小时跑完**
4. **LeRobot v3.0 数据格式迁移** + Multi-LerobotDataset 并行加载
5. **Rosbag 自动裁剪**(苏州 184 条 rosbag 按 arm_traj 话题裁剪无效段)
6. **数据质检**:话题缺失 / 掉频 / 漂移的自动检测

**数据资产**:**2900+ 条拆垛真机数据**,综合采数计划 3000+ 条

**面试话术**:"这是校招算法岗最容易被低估的能力。我在这 8 个月里最大的学习是:**一个 VLA 模型的上限,由它能见到的数据分布决定**。"

### 模块 2:VLA 模型训练 & 迭代 —— **【团队主干,深度参与】**

**模型矩阵**(我全部参与评测与数据准备,深度理解):

| 路线 | 代表模型 | 团队评测结论 |
|---|---|---|
| **模仿学习 IL** | ACT / iACT / comACT / DP / DP3 | ACT 是搬箱主力(单摞 80%);DP finetune encoder 后视觉伺服准;DP3 推理最快但 3D 语义弱 |
| **Flow Matching** | flow_matching(多头) | 推理 20Hz 最快,多头架构分离手臂和夹爪 |
| **端到端 VLA** | OpenVLA-OFT / RDT-1B / π0-fast / π0.5 / **Groot** | Groot(SigLIP + 全参微调)成为拆垛主力,4 类混训 85% |
| **RL 人机在环** | HIL-SERL / ConRFT | HIL-SERL 原地搬箱 50%,**dynamic BC loss 权重**是收敛关键 |
| **闭环迭代** | Dagger | 模型失败 → 补采 → 重训,显著提升鲁棒性 |

**里程碑成果**:
- 🎯 **Groot VLA 4 类箱体混训 85% 成功率**(白底 23/27)
- 🎯 **ACT 单摞搬运 80% 成功率**(绿框 3-4-5 层随机摆放)
- 🎯 **OpenVLA-OFT libero_spatial 98.8%**(超论文 97.6%)
- 🎯 **HIL-SERL 原地搬箱 50%**(小车基准 5min 示教 90%)

**我在其中的参与**:

- 压测评测集构建(in-dist / OOD 杂乱背景 / 金属重物 / 倾斜)
- **评测失败案例回流到增强管线**,形成"评测 → 增强 → 重训"闭环

**话术要点**:讲 Groot 85% 时要说清楚:

### 模块 3:部署 & 评测体系 —— **【参与】**

**推理加速链路**:
- .safetensors → **ONNX**(首次 4s 预热,后续 0.1s)→ **TensorRT**(**30Hz**,vs ONNX 10Hz 提升 3 倍)
- 动作时间优化:**11s → 9s**(通过 skip-chunk-ratio + 插值平滑 + 低通滤波,目标 7.5s)
- 推理平滑三件套:**chunk inpainting** + 低通滤波 + 线性插值 → 消除 chunk 边界跳变

**多机型适配**:
- Kuavo 双臂(主力)+ Kuavo 轮臂(MT-9 网口断连后迁 MT-7)
- IK 三方案评测:**一阶段数值 IK / 两阶段数值 IK / 解析解 IK**,根据精度-速度 trade-off 选型

**评测体系**:
| 评测维度 | Groot 成功率 |
|---|---|
| In-distribution(白底 4 类混训) | 85%(23/27) |
| OOD 金属重物 | 100%(1/1) |
| OOD 杂乱背景 | 83%(5/6) |
| OOD 白塑料 | 50%(2/4) |
| OOD 倾斜摆放 | 0%(0/2)← **下一步攻克目标** |

**工程化产出**:
- **kuavo-manip 轻量化 SDK v0.1.0**:打包本地 SDK / IK / msgs,数采脚本四合一

## 六、9 个"失败-修正"案例(面试亮点)

这是面试官最爱问的部分。每个案例都有明确的**现象 → 根因 → 修正**三段式:

1. **ACT 学不到视觉伺服** → 根因是 action space = cmd_pose 在时间轴上不连续 → 改回 cmd_vel + 少量人工干预(0530-0628)
2. **HIL-SERL 前期不收敛** → 专家数据 <30 条 + 没按 off-policy trick → >30 条专家 + **dynamic BC loss 加权** → 50%(0620-0823)
3. **DP 推理能力突然消失** → image encoder 对相机序号敏感 + OOD → Dagger 补采 + finetune encoder(1107-1114)
4. **iACT 走路稳但抬手卡顿** → 抬手前几秒相机 obs 几乎无变化 → 新增**胸部低视角相机** + 扩大 chunk_size(0906)
5. **ACT 推理 chunk 间跳变** → chunk 边界处预测突变 → **Inpainting + 低通滤波 + 插值平滑**(1114)
6. **拆搬混训模型混淆** → Task index / task_text 信息量不够 → **Groot 全参微调(VLA 路线)** → 85%(1114-1125)
7. **端到端推理<10Hz 达不到实时** → ONNX CPU 路径慢 → **TensorRT → 30Hz**(1128)
8. **数据中 TAG 干扰模型学习** → TAG 作为强特征过拟合 → **DINO+SAM+LaMa 自动擦除** 6h 全集(1024)
9. **背景/光线泛化不足(OOD 倾斜 0%)** → 真机数据背景单一 → **【我主导】Cosmos-transfer 合成 + SAM3+IC-light 备选**(1219)

> 第 9 个是你主导的,一定要讲得最详细。前 8 个是团队的,你讲 1-2 个作为"我理解全流程"的佐证即可。

## 八、后续方向(面试官追问"如果继续做会怎么做")

1. **高度泛化**:4322/4611 高度混训的 condition 网络设计(现在 Groot 看高度靠视觉,偶尔会混)
2. **Corner case 补采**:倾斜摆放 0% 是硬骨头,需要 Dagger 专门补采 + Cosmos-transfer 合成倾斜姿态
3. **3D 感知增强**:DP3 方向理解误差的根因是点云语义弱,可以引入 3D VLM(类似 3D-LLaVA / PointLLM 思路)
4. **推理加速**:9s → 7.5s 最后 20% 可以通过模型蒸馏(Groot → 小模型)拿到
5. **RL 规模化**:HIL-SERL 目前只做到原地搬箱 50%,走动场景需要更大的人工干预 buffer + Off-Policy trick
