## 三、业务背景

- **目标场景**:人形机器人在仓储 / 工业场景完成"搬箱"和"拆垛"两类核心任务,同时探索"**UMI 无本体采集**"作为未来低成本数据扩展的路径
- **硬件**:
  - Kuavo 人形(双臂)、Kuavo 轮臂(MT-9 / MT-7)
  - 4 摄视觉:头 + 胸部低视角 + 双肩(realsense / 奥比中光)
  - ZED-M 双目 + FoundationStereo 深度
  - **UMI 预研阶段**:手持夹爪 + 动捕系统(Vicon / OptiTrack)
- **四重挑战**:
  1. **长视地任务**(视觉伺服 → 走路 → 弯腰 → 抓取 → 起身 → 搬运 → 放下)
  2. **多构型泛化**(4311 / 4322 / 4611 / 4633 多箱体)
  3. **实时性**(Orin 嵌入式平台 30Hz)
  4. **数据扩展性**(真机采集成本高,是否能用 UMI 无本体打开空间?)

## 四、三阶段主项目架构

### 阶段 1:搬箱 ACT 时期 —— 采数工具链 & pytrees 框架【主导】

**时间**:2025.10月底 – 2025.11月底(~1 个月)

**背景**:入职时团队正在把 ACT 搬箱从研究原型推向工程落地,单摞 80% 成功率已达成,需要配套的采数工具和推理节点编排能力来支撑后续拆垛扩展。

**我的主导工作**:

#### 1.1 采数工具链实现与维护
- **四合一采数脚本**:统一原子技能 / VR 遥操 / 键盘 IK 微调 / 行为树辅助 四种采数模式的调用接口
- **自动 SSH 回传**:采完即时上传 H100 训练机,避免本地空间堵塞
- **数据质检**:话题缺失检测 / 掉频告警 / 时间戳漂移检查;避免脏数据进入训练集
- **天轨自动采数**:利用实验室天轨固定轨道,机器人自主走到箱前后才开始录制(早期对 ACT "学不会走路"问题的工程兜底)

#### 1.2 pytrees 推理节点封装
- **背景**:团队有多种推理策略(ACT / DP / flow_matching / Groot),但每个策略的推理入口、输入格式、输出格式都不一样。上层行为树(py_trees)编排时需要反复适配
- **我做的**:设计统一的 `ModelInferenceNode` 基类,每种策略实现对应子类,支持:
  - 行为树 tick 调用 → 加载模型 / 读取观测 / 推理 / 发布 action
  - 状态机:`IDLE / LOADING / INFERRING / PUBLISHING / DONE / FAILED`
  - 失败上抛:行为树可以选子节点切换策略(ACT 失败 → 切 Groot)
- **配合背景参考**:1128 周报提到"Groot 推理脚本封装为 pytree 节点",这条是我在做

**可讲的"我参与"部分**:
- ACT 搬箱后期的评测与小规模数据补采
- 旧版 ACT 模型在 pytrees 框架下的接入验证

### 阶段 2:拆垛 VLA 时期 —— 消融 + DAgger + 数据增强【主导 + 深度参与】

**时间**:2025.11月底 – 2026.02月底(~3 个月)

**背景**:搬箱成功后项目转向拆垛。拆垛任务更复杂(多构型箱体、堆叠层次、视觉遮挡),团队引入 π0.5 和 GR00T 1.5 作为 VLA 主力候选,最终选择 Groot。

**我的深度参与工作**:

#### 2.1 消融实验辅助(团队主导,我执行)

我辅助组里跑了多组消融,面试时可以主推两个最有把握的:

**(a) 数据规模消融** —— 最核心
- 从 2900+ 条真机数据中按比例抽样:500 / 1000 / 2000 / 2900 条
- 同样超参训 Groot,评测 in-dist 和 OOD 成功率
- 目的:找到**数据边际效用拐点**,指导后续采数预算分配
- 结论方向:数据从 1000→2000 提升显著,2000→2900 提升有限(说明已近饱和,OOD 失败不是靠更多同分布数据能解决的 → 为 SAM3+IC-light 方向提供了依据)

**(b) 相机配置消融**
- 4 摄(头+胸+双肩)vs 3 摄(去胸部)vs 3 摄(去单肩)
- 目的:评估**胸部低视角相机**对手-箱交互的实际贡献
- 结论方向:胸部低视角对"抬手切换"阶段成功率提升明显(对应 0906 周报的已知问题)

**(c) 数据增强 ratio 消融**(和 2.3 联动)
- 原数据 vs 原数据 + 25% SAM3+IC-light 合成 vs 原数据 + 50% 合成
- 目的:找到合成数据占比的甜点

#### 2.2 DAgger 失败恢复【我主导】

**问题**:Groot 在白底 in-dist 达到 85%,但 OOD 倾斜摆放 **0%**、白塑料 **50%**、杂乱背景 83%。这些 corner case 不是靠更多同分布真机数据能解决的。

**我的解法**:
1. **建立失败评测集**:从每次压测中抽取失败 episode,按失败模式分类(感知失败 / 规划失败 / 控制失败)
2. **DAgger 干预回流**:
   - 把模型部署到失败场景
   - 人通过 VR 遥操或键盘接管失败 trajectory
   - 干预数据自动录制,打专属标签(intervention = True)
   - 进入下一轮训练的 buffer,权重可调(类似 HIL-SERL 的 BC loss 加权)
3. **其他数据源扩充**:
   - **仿真反例采集**:在 IsaacLab 中故意构造 corner case 场景
   - **合成数据增强**(与 2.3 联动)
   - **主动采样**:下次采数优先采已知失败模式的场景

**关键工程选择**:
- DAgger 和纯 BC 的 trade-off:纯 BC 失败恢复收敛慢,DAgger 干预数据密度高但可能 distribution shift
- 我采用"**混合缓冲**"策略:原数据占主,干预数据占 10-30%,避免过度 overfit 到 corner case

#### 2.3 数据增强:SAM3+IC-light 落地【我主导】

**调研阶段**:两条候选路线

**(a) Cosmos-transfer**(NVIDIA 视频扩散模型)
- 优点:整段视频背景替换,时序一致性好
- 缺点:**推理慢 + 算力需求大 + 生成质量波动**(有时箱子形变 / 消失)
- **结论**:我评估后放弃。受限于时间紧张和 H100 算力资源分配,Cosmos 不现实

**(b) SAM3 + IC-light**(最终落地方案)
- **SAM3**:对每帧图像分割前景(机器人 + 箱子)
- **IC-light**:对前景保留的情况下,对背景重新光照渲染
- 优点:**更轻量**(单卡可跑)、**生成质量更可控**、**可批量处理现有 2900+ 数据**
- 可以与 SAM3 组合增加多样性:前景 mask 反向处理 → 背景完全替换

**落地效果**:
- 目标:把 OOD 杂乱背景 83% → 90%+,倾斜 0% 往上拉
- 由于离职时间点原因,量化 AB 测试留给后续工作

### 阶段 3:UMI 无本体采集预研【主导】

**时间**:2026.03 – 2026.04月底(~2 个月)

**背景**:真机遥操采数成本高(1 个 episode 需 30 秒-1 分钟+标注),如果能用 UMI(手持夹爪 + 相机)让任何人都能采数据,就能把数据规模从千级推到万级。团队把这个预研方向交给我负责。

**我的主导工作**:

#### 3.1 UMI 方案初期构建
- **参考 Cheng Chi et al. RSS 2024** "UMI: Universal Manipulation Interface"
- 核心架构:手持夹爪 + 硬件相机(GoPro / realsense),采集人类手部动作和夹爪开合
- 采集到的相机位姿 + 夹爪状态 → 转换为机器人 end-effector 动作

#### 3.2 场地优化:放弃 ORB-SLAM → 动捕
- **原 UMI 方案用 ORB-SLAM**:基于单目/双目相机视觉 SLAM 恢复相机位姿
- **实测问题**:
  - 训练场墙面 / 地面特征少(白墙),SLAM 特征点丢失
  - 手持夹爪快速运动时,ORB-SLAM 跟踪失败率高
  - 精度不稳定,有时一条 trajectory 中途漂移
- **我的替代方案:采用动捕系统(Vicon / OptiTrack)**
  - 在手持夹爪和场地四角贴 marker
  - 动捕系统毫米级精度实时输出位姿
  - 优势:**确定性位姿,不依赖视觉特征**
  - 代价:**受动捕覆盖范围限制**(一般 5m × 5m 左右),不适合大范围操作

#### 3.3 本体实时映射
- 坐标系转换:UMI 夹爪坐标系 → 机器人 base 坐标系 → end-effector 坐标系
- 动力学适配:UMI 采集速度可能超过机器人最大关节速度 → 需要平滑和降采样
- 视角差异处理:UMI 相机固定在夹爪上,机器人相机在头部/胸部 → 视角 gap

#### 3.4 UMI vs VR 方案对比(关键洞察)

| 维度 | UMI(手持夹爪 + 动捕) | VR 全身遥操(Pico / Quest3 / MetaVR) |
|---|---|---|
| 可操作性 | **方便,长期采集不累** | **过于沉重,1-2 小时就疲劳** |
| 场地限制 | **受动捕范围限制**,适合短距离精细操作 | **不受场地限制**,长距离可用 |
| 数据质量 | 动作连续,位姿精准(动捕级) | **数据传输不均匀,断断续续**(VR 硬件瓶颈) |
| 采数效率 | 高(任何人能采) | 低(需专业遥操员) |
| embodiment gap | **大**(见 3.5) | **小**(VR 直接驱动真机) |

#### 3.5 关键负面结论:Lingbot-VLA 30%

**实验**:
- 用 UMI + 动捕采了 **1000 条"放置小零件到盒子内部"** 任务数据
- 用 **Lingbot-VLA**(业内 VLA 基线模型)训练
- 部署到真机评测

**结果**:
- **成功率只有 30% 出头**
- 失败模式分析:
  - 视角不一致:UMI 相机位于夹爪,机器人相机位于头部/胸部
  - 夹爪形态差异:UMI 夹爪几何与机器人末端夹爪不完全一致
  - 精细动作要求高:放置小零件需要毫米级精度,小的 embodiment gap 会被放大

**结论**:
- 无本体采集的初衷(零成本数据扩展)在**精细操作任务**上不成立
- UMI 适合采**粗精度 / 大范围操作**数据(比如拾取、搬运),但对**精细操作**不如真机遥操
- **建议公司后续 UMI 投入只放在粗精度任务,精细任务仍然用真机 VR 遥操**
