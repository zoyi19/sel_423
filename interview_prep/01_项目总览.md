## 四、三阶段主项目架构

### 阶段 1:搬箱 ACT 时期 —— 采数工具链 & pytrees 框架【主导】

**时间**:2025.10月底 – 2025.11月底(~1 个月)

**背景**:入职时团队正在把 ACT 搬箱从研究原型推向工程落地,单摞 80% 成功率已达成,需要配套的采数工具和推理节点编排能力来支撑后续拆垛扩展。

**我的主导工作**:

#### 1.1 采数工具链实现与维护

**四件套**:四合一脚本 → 归档到数据存储机 → 数据质检 → **bag → LeRobot v3.0 转换器**(`bag2lerobot.py`,~1400 行,我亲手写的核心产出)。下面详细讲转换器里的技术决策,因为这是面试最容易被刨根问底的部分。

##### (1) 四合一采数脚本
统一原子技能 / VR 遥操 / 键盘 IK 微调 / 行为树辅助 四种采数模式的调用接口,对上层透明。

##### (2) 数据归档链路
采完 rosbag 自动归档到**专用数据存储机**(非 H100 训练机,实时上传会占用训练带宽);训练时再按需从数据机统一拉取、转 LeRobot v3.0 格式。

##### (3) 数据质检
话题缺失检测 / 掉频告警 / 时间戳漂移检查;避免脏数据进入训练集。

##### (4) 天轨自动采数
利用实验室天轨固定轨道,机器人自主走到箱前后才开始录制(早期对 ACT "学不会走路"问题的工程兜底)。

##### (5) bag → LeRobot v3.0 转换器 —— 核心技术细节

**(a) 多 ROS topic 时间戳对齐 + Forward Fill**
- 从 rosbag 读取 N 个 topic 的异步消息(图像 / 关节 / IMU / MPC target / 步态 / 手臂轨迹),扫一遍统计 [min_ts, max_ts]
- 构造均匀时间轴 `time_grid = np.arange(min_ts, max_ts+dt, dt)`,**dt=0.1s(10Hz 采样)**
- 每个 topic 对 time_grid 做 **forward fill**,但第一帧做特殊处理:
  - 第一帧:用**第一个有效消息的真实值**(避免 delta 计算时的冷启动大跳变)
  - 后续无消息帧:**填零**(避免误用未来消息)
  - 有历史消息:正常向前填充
- **为什么这样设计**:delta action 对第一帧非常敏感,如果第一帧填零而第二帧是真实位姿,delta 会瞬间巨大,模型学不好

**(b) 混合 Action Space 设计(面试亮点)** —— **这是我做过的最有工程味的决策**

action 共 **24 维**,但分成三段不同处理:

| 段 | 维度 | 编码方式 | 为什么这样 |
|---|---|---|---|
| **COM(质心)** | 9 维 | **delta (相对增量)**:`(current - prev) / dt` | 走路是长距离连续任务,absolute 位置会让模型学坐标系,泛化差;delta 学"位移率"更可泛化 |
| **Arm(双臂关节)** | 14 维(左 7 + 右 7) | **absolute(弧度)** | 关节必须绝对精准,delta 会累积误差,抓取会偏 |
| **Gait(步态)** | 1 维 | **discrete**:walk=1 / stance=2 | 本质是分类变量,标量编码让模型容易学 |

> 面试时用一句话概括:**"不是所有 action 都该用同一种表示,哪一段用 delta / abs / discrete,要看任务本身的几何性质"**。

**(c) COM 的 9D 表示:位置 3 + 6D 旋转**
- 位置:`(dx, dy, dz)` delta
- 旋转:**不用欧拉角,不用四元数,用 Zhou et al. 2019 的 6D 旋转表示**(旋转矩阵前两列,按 Fortran order 列展开)
- **为什么 6D?**
  - 欧拉角有万向锁、不连续
  - 四元数是 double cover(q 和 -q 表示同一个旋转,回归任务不友好)
  - **6D 表示是连续、无奇点的,对神经网络回归最友好**
- delta 旋转:`R_delta = R_current · R_prev^T`,再转回 6D
- 从 6D 重构回 3x3 用 **Gram-Schmidt 正交化**,并加 `1e-8` epsilon 防止 col 范数 0 时除零

**(d) State 里放 Previous Delta(关键巧思)**
- observation.state = **38 维** = `[14 arm + 3 lin_acc + 3 ang_vel + 9 com_6d(当前) + 9 prev_delta(上一帧)]`
- **为什么把"上一帧的 delta"放进 state?**
  - 让模型推理时能看到"刚才动了多少",等价于给模型一个**短期记忆**
  - 消除 chunk 边界的不连续感,策略学到的动作更平滑
  - 这是 comACT 的 well-known trick,我在自己的转换脚本里显式实现

**(e) 积分面积保持的 Delta 平滑化**
- delta 序列过 Savitzky-Golay 滤波(window=11, polyorder=3),让动作更平滑
- **但有一个坑**:纯滤波会改变"总位移"(积分面积)→ 机器人会走错地方
- **我的解法**:平滑后计算 `scale_factor = sum(original) / sum(smoothed)`,按维度缩放
- 最后验证每个位置维度的相对误差 < 1%
- **这一步体现了"平滑"和"动作正确性"之间的务实权衡**,是工程+理论结合的典型例子

**(f) 图像特征预提取加速训练**
- 3 摄图像原样存是磁盘爆炸 + 训练 IO 瓶颈
- 我用 **ResNet18(ImageNet pretrained)+ FrozenBatchNorm2d**,取 `layer4` feature map 离线预提取
- 图像先 resize 到 240×320 → normalize → backbone → feature,**存为 hex-encoded binary bytes**
- 训练时直接加载 feature,不过 encoder
- **对应 1018 周报**:这个预提取让 80 条数据 2 小时跑完 250 epoch(相比原生 online encoding 提速数倍)

**(g) 其他工程细节**
- `arm_traj` 原始数据是**度**,需要 `np.deg2rad()` 转弧度(踩过一次把度值当弧度训 → 模型崩)
- LeRobot v3.0 features schema 严格,binary 类型必须 hex 编码成 ASCII bytes
- 多 bag 批处理:每个 bag 对应一个 episode,可以**无缝支持后续 Multi-LerobotDataset 并行加载**
- 支持 `--overwrite` / `--disable-smoothing` / `--use-image-features` / `--dataset-mode video/image` 等配置
- 10 线程 image_writer + 5 进程并行写盘,吞吐压榨到硬件上限

#### 1.2 pytrees 推理节点封装

**设计思想**:**把"策略接入"变成一个"按标准格式实现 Adapter"的工作**。上层行为树只依赖两个标准 schema(`ObservationFrame` / `ActionChunk`),任何新策略只需写一个 Adapter 把自己的入口对齐到这两个 schema 即可,上层代码零改动。这是典型的**模板方法 + Adapter 模式**。

**问题根源:为什么需要这层封装?**

团队多种推理策略的入口格式差异非常大:

| 策略 | 输入 | 输出 | action space | 特殊点 |
|---|---|---|---|---|
| **ACT** | multi-cam images + robot state(单帧) | chunk of shape (32, action_dim) | joint_pos / eef_pose | 单帧输入,一次出 chunk |
| **DP / DP3** | n_obs_step 历史帧(常 2–17 帧)+ state | horizon 步动作(默认 8) | eef_pose / joint_pos | **需要维护 obs queue**,DDPM 多步 denoise |
| **flow_matching** | 同 DP,但 flow 推理步数更少 | **multi-head**:arm + gripper 两路输出 | eef_pose / joint_pos + gripper | **推理 20Hz**,多头要分开解码 |
| **Groot / π0.5** | multi-cam + state + **language instruction** | DiT head 连续 chunk | 通常 joint_pos | **吃 TensorRT engine**,首次加载 4s |

**不同策略的入口差异具体体现在**:
1. **观测拼装方式**:ACT 只吃当前帧;DP 需要历史 n_obs_step 帧的时间窗;Groot 还要语言指令
2. **推理调用**:`policy.select_action(obs)`(ACT)、`policy.predict_action(obs_dict)`(DP)、`policy.forward(obs, text, state)`(Groot)、TensorRT bind I/O(Groot 加速路径)
3. **输出动作语义**:joint_pos / eef_pose / cmd_vel / cmd_pose 混杂;多头模型夹爪单独出一路
4. **发布频率**:ACT/DP <10Hz,flow_matching 20Hz,Groot TensorRT 30Hz
5. **加载耗时**:ACT / DP 的 pth 秒级就能加载;Groot TensorRT engine 首次**预热 4s**;π0.5 更久

**我的解法:两个标准 schema + Adapter 层**

**输入侧标准格式 `ObservationFrame`**(所有策略接受同一个输入数据类,**与 `bag2lerobot.py` 里的 observation.state 38 维字段结构对齐**):
```python
@dataclass
class ObservationFrame:
    images: Dict[str, np.ndarray]   # {"head": HxWxC, "left_wrist": ..., "right_wrist": ...}
    state:  np.ndarray              # 38 维: [14 arm + 3 lin_acc + 3 ang_vel + 9 com_6d + 9 prev_delta]
                                    # 训练时怎么拼,推理时就怎么拼,避免 distribution shift
    task_text: Optional[str]        # language instruction(Groot / π0.5 才用,其他忽略)
    timestamp: float                # 时间对齐用
```
> **关键约束**:推理时的 state 拼装必须与训练时一致(`bag2lerobot.py` 决定的 38 维顺序)。Adapter 负责把机器人实时话题(dof_state / IMU / com / MPC target)拼成同样的 state 向量,这是 online serving 里最容易踩的坑。

**输出侧标准格式 `ActionChunk`**(与训练 action 24 维对齐):
```python
@dataclass
class ActionChunk:
    actions: np.ndarray             # (chunk_size, 24):9 COM delta + 14 arm abs + 1 gait
    action_space: str               # 标注每段的语义:"com_delta" / "arm_abs_rad" / "gait_discrete"
                                    # publisher 按语义分发到对应 ROS topic
    execute_freq: float             # 10Hz(配合训练 dt=0.1s)
    gripper:  Optional[np.ndarray]  # 多头模型把夹爪单拎出来(flow_matching 用)
    metadata: Dict                  # 置信度 / chunk 序号 / 推理 latency
```
> **COM delta 特殊处理**:推理输出是 delta,publisher 必须**累加回 absolute**才能发给 MPC。并且要维护 `prev_delta` 状态(下一帧推理时写回 `ObservationFrame.state` 的最后 9 维)。这个闭环是 Adapter 层必须处理的,上层不关心。

**策略 Adapter 写法(每个策略一个子类,~50 行代码)**:
```
ACTAdapter          → 取单帧 images + state,调 select_action,返 joint_pos chunk
DPAdapter           → 维护 obs_queue(deque maxlen=n_obs_step),调 predict_action,返 eef_pose
FlowMatchingAdapter → 类似 DP,但拆 arm / gripper 两个 head 到 ActionChunk
GrootAdapter        → 加语言指令 + SigLIP preprocess + TensorRT bind I/O,返 joint_pos
```

**上层 `ModelInferenceNode`(py_trees.Behaviour 子类)**:只与两个 schema 交互,不关心底层策略
- `IDLE → LOADING`:行为树第一次 tick 时触发加载(异步线程,不阻塞 tick)
- `LOADING → INFERRING`:加载完成后读取 blackboard 上的 `ObservationFrame`
- `INFERRING → PUBLISHING`:拿到 `ActionChunk` 交给 action publisher 按 `execute_freq` 发布
- `PUBLISHING → DONE` 或 `FAILED`:chunk 执行完 / 超时 / 异常 → 行为树回传 SUCCESS / FAILURE
- **失败上抛**:行为树 `Selector` 父节点据此切换到备用策略(ACT 失败 → 切 Groot)

**为什么这样设计?三条核心好处**

1. **策略接入零上层改动**:新模型进来,只写一个 Adapter,`ModelInferenceNode` 和行为树编排完全不动
2. **配置驱动切换**:运行时 YAML 改 adapter 类型 + pth 路径 → 立刻换模型,不需要重启
3. **统一错误处理**:CUDA OOM / 话题缺失 / 推理超时都在 Adapter 里 catch 后返回 FAILED,上层 Selector 接住

**踩过的坑**(面试可讲):
- TensorRT engine 首次加载 4s 阻塞 tick → 改成**异步加载**,加载期间 tick 返回 RUNNING
- 多摄时间戳对齐:3 摄(头+双腕)到达时间不一致 → 加了 buffer + 时间戳校验,**最旧一帧超过 100ms 就丢弃重拼**
- ROS 话题发布频率与 `execute_freq` 不一致 → publisher 单独跑一个定时器线程,不阻塞推理

**配合背景**:1128 周报"Groot 推理脚本封装为 pytree 节点"是团队里最早用 pytrees 封装的痕迹,我在此基础上把它扩展成**支持多策略的通用框架**,不再是"为 Groot 写一个一次性节点"。

**可讲的"我参与"部分**:
- ACT 搬箱后期的评测与小规模数据补采
- 旧版 ACT 模型在 pytrees 框架下的接入验证1

#### 2.1 消融实验辅助(团队主导,我执行)

- 目的:找到合成数据占比的甜点

#### 2.2 DAgger 失败恢复【我主导】

**问题**:Groot 在白底 in-dist 达到 85%,但 OOD 倾斜摆放 **0%**、白塑料 **50%**、杂乱背景 83%。这些 corner case 不是靠更多同分布真机数据能解决的。

**我的解法**:
1. **建立失败评测集**:从每次压测中抽取失败 episode,按失败模式分类(感知失败 / 规划失败 / 控制失败)
2. **DAgger 干预回流**:
   - 把模型部署到失败场景
   - 人通过 VR 遥操或键盘接管失败 trajectory
   - 干预数据自动录制,打专属标签(intervention = True)
   - 进入下一轮训练的 buffer,权重可调(原数据 70-90%,干预数据 10-30%)
3. **其他数据源扩充**:
   - **仿真反例采集**:在 IsaacLab 中故意构造 corner case 场景
   - **合成数据增强**(与 2.3 联动)
   - **主动采样**:下次采数优先采已知失败模式的场景

**先讲为什么没选 RL**(对照 π0.6 的 RL 路线):
- RL fine-tune VLA 是另一种思路(类似 π0.6),让模型从 reward 学失败恢复
- **但真机 RL 不现实**:PPO 是 on-policy(数据用完即丢,真机数据贵浪费不起)、reward 难设计(拆垛任务定义"成功"很模糊,容易 reward hacking)、样本效率低(10^7+ 步)
- 业界主流是"IL pretrain + 仿真 RL fine-tune"或离线 RL,不会在真机直接 RL
- **DAgger 范式与 IL 训练对齐**,工程改动小,数据安全可控 → 选 DAgger

**关键工程选择**:
- DAgger 和纯 BC 的 trade-off:纯 BC 失败恢复收敛慢,DAgger 干预数据密度高但可能 distribution shift
- 我采用"**混合缓冲**"策略:原数据占主,干预数据占 10-30%,避免过度 overfit 到 corner case

**🪝 拟合悖论(钩子,讲到这里要主动停一下引导追问)**:
- 业界要求 95+ 成功率,但**正常训练规模训不出 95+**,实际是"过拟合到某种程度才能达到"
- DAgger 提供这个"针对性过拟合"的路径,**但干预数据数量永远不够**:
  - 失败模式是 long tail,估计 100+ 个失败模式
  - 每个至少 5-10 条干预数据 → 总共 2500-5000 条
  - 每条 30 秒人工接管 + 标注 → 5000 条 = 41+ 小时纯人工,1-2 周成本
  - 真实失败模式远不止 100 个 → DAgger 永远补不完
- 我有 4 个想过但没验证的方向(weight decay / curriculum / 评测分两套 / 失败模式聚类)— **这是面试官追问时展开的弹药**

#### 2.3 数据增强:SAM3+IC-light 落地【我主导】

**调研阶段**:三条候选路线(由浅到深逐个排除)

**(a₀) OpenCV 传统增强** —— 评估后放弃
- 工具:亮度 / 对比度 / 翻转 / 裁剪 / 高斯噪声 / 颜色抖动
- 致命问题:**像素层增强,改不了场景结构和语义**(改背景颜色 ≠ 改成"杂乱仓库背景")
- 真机 OOD 测试**几乎无提升**,舍弃

**(a) Cosmos-transfer**(NVIDIA 视频扩散模型)—— 评估后放弃
- 优点:整段视频背景替换,时序一致性好,理论上效果最好
- 缺点:**4090 跑不动,需要 H100 + 单视频生成数分钟 + 生成质量波动**(有时箱子形变 / 消失)
- **结论**:我评估后放弃。受限于时间紧张和 H100 算力资源分配,Cosmos 不现实

**(b) SAM3 + IC-light** —— 最终落地方案
- **SAM3**:对每帧图像分割前景(机器人 + 箱子)
- **IC-light**:对前景保留的情况下,对背景重新光照渲染
- 优点:**更轻量(单 4090 可跑)**、**生成质量更可控**、**可批量处理现有 2900+ 数据**
- **量化产出**:**单个 3 分钟操作视频 → 10 分钟内生成 12 个不同背景的视频**
- 可以与 SAM3 组合增加多样性:前景 mask 反向处理 → 背景完全替换

**落地效果**:
- 目标:把 OOD 杂乱背景 83% → 90%+,倾斜 0% 往上拉
- 由于离职时间点原因,量化 AB 测试留给后续工作

**🪝 钩子:Cosmos 在更好显卡下值得重新评估** — SAM3+IC-Light 改不了场景结构和视角,Cosmos 能改;有 8×H100 的话,值得做"原数据 / +SAM3 / +Cosmos / +两者组合"的对比实验

### 阶段 3:UMI 无本体采集预研【主导】

**时间**:2026.03 – 2026.04月底(~2 个月)

**背景**:真机遥操采数成本高(1 个 episode 需 30 秒-1 分钟+标注),如果能用 UMI(手持夹爪 + 相机)让任何人都能采数据,就能把数据规模从千级推到万级。团队把这个预研方向交给我负责。

**我的主导工作**:

#### 3.1 UMI 方案初期构建
- **参考 Cheng Chi et al. RSS 2024** "UMI: Universal Manipulation Interface"
- 核心架构:手持夹爪 + 硬件相机(GoPro / realsense),采集人类手部动作和夹爪开合
- 采集到的相机位姿 + 夹爪状态 → 转换为机器人 end-effector 动作

#### 3.2 场地优化:放弃 ORB-SLAM → 动捕
- **原 UMI 方案用 ORB-SLAM**:基于单目/双目相机视觉 SLAM 恢复相机位姿
- **实测问题**:
  - 训练场墙面 / 地面特征少(白墙),SLAM 特征点丢失
  - 手持夹爪快速运动时,ORB-SLAM 跟踪失败率高
  - 精度不稳定,有时一条 trajectory 中途漂移
- **我的替代方案:采用动捕系统(Vicon / OptiTrack)**
  - 在手持夹爪和场地四角贴 marker
  - 动捕系统毫米级精度实时输出位姿
  - 优势:**确定性位姿,不依赖视觉特征**
  - 代价:**受动捕覆盖范围限制**(一般 5m × 5m 左右),不适合大范围操作

#### 3.3 本体实时映射
- 坐标系转换:UMI 夹爪坐标系 → 机器人 base 坐标系 → end-effector 坐标系
- 动力学适配:UMI 采集速度可能超过机器人最大关节速度 → 需要平滑和降采样
- 视角差异处理:UMI 相机固定在夹爪上,机器人相机在头部/胸部 → 视角 gap

#### 3.4 三方对比:真机遥操 / VR / UMI

**先讲 VR 为什么先被排除**(预研最早决定的事):
- 设备**沉重,1-2 小时疲劳**,不能长期采集
- **采集到的数据不平衡**(某些姿态/动作分布偏),**不能直接训练**
- VR 数据传输还会断断续续 → 综合评估后**先排除 VR**,再聚焦在真机 vs UMI 的取舍

| 维度 | 真机遥操(基线) | VR 全身遥操(Pico/Quest3/MetaVR) | **UMI(手持夹爪 + 动捕,选)** |
|---|---|---|---|
| 可操作性 | 中 | **过于沉重,1-2h 疲劳** | **方便,长期采集不累** |
| 场地限制 | 不限 | 不限 | **受动捕范围限制(~5m × 5m)**,适合短距离精细操作 |
| 数据质量 | **最好(直接训机器人)** | **传输不均匀,数据不平衡,不能直接训练** | 动作连续,位姿精准(动捕级) |
| 采数效率 | 低(每条 30s+) | 低(需专业遥操员) | **高(任何人能采)** |
| embodiment gap | **0** | 小(VR 直接驱动真机) | **大(精细任务尤甚)**(见 3.5) |
| 适用任务 | 全部 | 演示 / 小规模 | **粗精度大量数据** |

**关键洞察**:**三种方式不是互相替代,是互补**(粗精度大量数据 → UMI;精细 → 真机)。

#### 3.5 关键负面结论:Lingbot-VLA 30%

**实验**(方案我主导设计,采集与一位同事合作完成):
- 用 UMI + 动捕采了 **1000 条"放置小零件到盒子内部"** 任务数据
- 用 **Lingbot-VLA**(开源 VLA 模型,与公司有合作关系)训练
- 部署到真机评测

**结果**:
- **成功率只有 30% 出头**
- 失败模式分析:
  - 视角不一致:UMI 相机位于夹爪,机器人相机位于头部/胸部
  - 夹爪形态差异:UMI 夹爪几何与机器人末端夹爪不完全一致
  - 精细动作要求高:放置小零件需要毫米级精度,小的 embodiment gap 会被放大

**结论**:
- 无本体采集的初衷(零成本数据扩展)在**精细操作任务**上不成立
- UMI 适合采**粗精度 / 大范围操作**数据(比如拾取、搬运),但对**精细操作**不如真机遥操
- **建议公司后续 UMI 投入只放在粗精度任务,精细任务仍然用真机 VR 遥操**
