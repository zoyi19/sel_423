## 主题 1:VLA 架构 —— OpenVLA / RDT / π0 / π0.5 / Groot 的差异

### Q1.1 "你们尝试过哪些 VLA 模型,为什么最后选 Groot?"

**核心答案框架**:
1. 我们做过 4 类 VLA 的对标:**OpenVLA-OFT、RDT-1B、π0-fast、Groot、π0.5**
2. 选型维度:**数据需求、推理速度、多任务能力、实机部署友好度**
3. **Groot(SigLIP vision encoder + 全参微调)**最终胜出,原因:
   - 多任务混训能力强(4 类箱体混训 85%),OpenVLA-OFT 在单摞任务 98.8% 但多任务会混
   - 推理可以通过 ONNX → TensorRT 压到 **30Hz**,RDT-1B 和 π0-fast 推理慢
   - 数据规模 2900+ 条真机够用,不像 UniVLA 需要 8×A100 或 RDT-1B 需要 150K step
4. 踩过的坑:RDT-1B 和 π0-fast 我们拿 50 条数据训,只学到"走向箱子"没学到抬手,说明这些大 VLA 对数据规模非常敏感

### Q1.2 "Groot 和 OpenVLA 架构上主要区别?"

**核心要点**(只讲你有把握的):
- **OpenVLA**(OpenVLA-OFT):基于 Llama + DINOv2 + SigLIP 视觉,action 离散化成 token 做自回归输出
- **Groot**(NVIDIA GR00T N1/N1.5):双系统架构(快系统 + 慢系统),视觉用 SigLIP,action 用 Diffusion Transformer(DiT)head 做连续动作生成
- 差异核心:
  - OpenVLA 离散 action token,Groot 连续 action via DiT
  - Groot 的双系统更适合长视地任务(慢系统做规划,快系统做执行)
  - OpenVLA 开源生态更好,Groot 工程更完整

**周报可引用数字**:
- OpenVLA-OFT:libero_spatial 复现 98.8%(论文 97.6%),单 GPU 30K step
- Groot N1.5:4 类箱体混训 85%(23/27),H100 多卡,400 epoch

### Q1.3 "π0 和 π0.5 区别?"

**核心要点**:
- **π0**(Physical Intelligence 2024 年论文):PaliGemma-3B + flow matching action head,在 OXE 等大规模数据上预训练
- **π0.5**(2025 年升级):更大规模预训练 + 更好的跨机器人迁移能力 + 更强的指令理解
- 我们实测:π0.5 在 LeRobot v3.0 上 H100 10h 跑 3K step,尚未达到可用水平(数据规模不够)

### Q1.4 "π0 的 flow matching 和 Diffusion Policy 的 DDPM 有什么区别?"

**核心要点**:
- Flow Matching:学从噪声到动作的**连续流(vector field)**,ODE 求解,理论上任意步数推理
- DDPM:学**离散时间步的反向扩散**,SDE 求解,通常 100+ 步推理
- 工程上:Flow Matching 推理步数可以压到 **10 步以内**,所以我们的 flow_matching 实现可以做到 **20Hz**;DP 通常 <10Hz
- 参考论文:Pi0(2024)、RF(Rectified Flow,2022)

**坦诚话术**:
> "Flow Matching 的理论细节我只看到 rectified flow 和 conditional flow matching 这两个层面,如果要讲到具体的 SDE/ODE 推导我只能讲个大概。组里汤赟熙在这块比我深入。"

## 主题 2:ACT 原理与 chunk size

### Q2.1 "讲一下 ACT 的核心设计?"

**核心答案**:
- **ACT = Action Chunking Transformer**(Aloha 那篇论文)
- 关键设计:
  1. **Action Chunking**:预测未来 K 步动作而不是单步,**K 通常取 16-100**
  2. **Temporal Ensemble**:每一步执行时,对过去多次推理结果做 ensemble(加权平均)
  3. **CVAE 结构**:encoder 接过去观测 + 动作序列 → latent z → decoder 生成动作
- 我们项目里 ACT 做到单摞 80% 成功率

### Q2.2 "chunk size 怎么选?太大太小有什么问题?"

**核心要点**:
- **太大**(我们试过 100)→ 一次预测太长,覆盖任务阶段切换点,模型学不准;**容易过拟合到训练轨迹**
- **太小**(比如 8)→ ensemble 的"预测视野"小,平滑度差,边界跳变明显
- 我们踩的坑:chunk_size=100 时搬箱效果反而比 32 差,说明长视地任务要谨慎
- 最终:搬箱用 **chunk_size=32 到 64** 较好

**周报引用**:"我们在 0912 前后对 chunk_size 做过对比,size=100 时搬箱精度明显差于 size=32"

### Q2.3 "chunk 边界会有跳变,你们怎么处理?"

**核心答案**:
- 现象:两个 chunk 边界处预测动作突变,实物表现为"抖动 / 冲过头"
- 我们的**推理平滑三件套**:
  1. **Inpainting**:chunk 重叠部分用前一 chunk 的尾 + 当前 chunk 的头做加权融合
  2. **低通滤波**:对 delta action 用 Savitzky-Golay 滤波器(时间戳 Δt=0.25s)
  3. **插值**:chunk 内对 target 做线性插值
- 这三件套在 comACT 上实测把实物抖动基本消除(1114 周报)

### Q2.4 "ACT 和 DP 你们怎么选?"

**核心答案**:
- **ACT**:工程简单,数据少时更容易训出可用模型 → **搬箱主力**(80% 单摞)
- **DP**:视觉伺服好,但推理慢(<10Hz),chunk 跳变更严重;finetune encoder 后在 y 方向上更准
- 选型原则:**长视地任务(搬 + 走 + 放)用 ACT;短视地精细操作(原地抓取)可以用 DP**

## 主题 3:Diffusion Policy / DP3

### Q3.1 "DP 和 DP3 区别?"

**核心要点**:
- **DP**:2D 图像 + CNN encoder + DDPM denoise 预测动作(Columbia 2023)
- **DP3**:输入改成 3D 点云 + PointNet encoder(CMU 2024)
- 优势:DP3 对视角 / 光照鲁棒性好于 DP,推理速度最快
- 劣势:我们实测 **DP3 方向理解有误差**(1219 周报),3D 语义还不够强;噪声鲁棒性也不理想(训练噪声 0.02,测试 0.00/0.01/0.02 下成功率 0.42/0.34/0.32)

### Q3.2 "DP 的 horizon 和 n_obs_step 怎么选?"

**核心要点**:
- **horizon**:预测未来多少步,我们常用 8
- **n_obs_step**:输入多少帧历史观测,n_obs_step=2 时推理启动快但视觉歧义多;n_obs_step=17 时启动慢但歧义少
- Trade-off:**推理速度 vs 启动延迟 vs 视觉歧义**
- 我们的经验:先大再小逐步调,DP 搬箱 horizon=8 + n_obs_step=17 相对好用

### Q3.3 "DP 推理能力突然消失是什么情况?"

**核心答案**(踩坑案例):
- 现象:某些初始位置推理可以成功,但推理多次后能力消失
- 根因分析:
  1. **Image encoder 对相机序号敏感**(不同 camera key 顺序被强编码)
  2. **OOD 分布**(位置落在训练分布边缘)
- 解法:Dagger 补采边界数据 + finetune encoder(解决 y 方向偏差)
- 反思:DP 在 OOD 上更脆弱,这也是我们最终没选 DP 做主力的原因之一

## 主题 4:HIL-SERL / ConRFT / RLPD —— 人机在环 RL

### Q4.1 "HIL-SERL 是什么?你们怎么用的?"

**核心答案**:
- HIL-SERL = **Human-In-the-Loop SERL**(Sergey Levine 组 2024 年),在 SERL(Sample-Efficient RL)基础上加入人工干预
- 核心机制:
  1. **Off-policy RL(SAC 为主)**
  2. **人工干预 buffer**:训练时人可以随时接管,干预样本进专用 buffer
  3. **BC loss 混合**:early-stage 用 BC loss 加速收敛,后期降权让 agent 自主探索
- 我们的成果:
  - 小车仿真:**5 分钟示教达到 90%**(vs 纯探索 40 分钟)
  - Kuavo 原地搬箱:**50% 成功率**(dynamic BC loss 权重是关键)

### Q4.2 "HIL-SERL 前期为什么不收敛?"

**核心答案**(踩坑):
- 现象:官方 Franka pick & place 能复现,但我们自己的 Kuavo 一开始完全不收敛
- 根因:
  1. **专家数据 <30 条**(论文建议至少 30+)
  2. **没按 off-policy 的标准 trick 做**(obs 归一化、熵初始化、梯度更新频率对齐等)
- 解法:
  1. 补采专家数据 > 30 条
  2. **dynamic BC loss 权重**:前期加大 BC loss 占比,让 agent 先模仿专家;后期降到 0 让 agent 自主探索
  3. 修正梯度更新频率、obs 归一化、熵初始化值
- 结果:从"不收敛"到"50% 原地搬箱"

### Q4.3 "ConRFT 和 HIL-SERL 有什么区别?"

**核心要点**:
- **ConRFT**:Continuous Reinforcement Fine-Tuning,把预训练的 IL policy 作为起点,RL 微调
- **HIL-SERL**:从零开始 RL 训练,人工干预加速
- 差异:
  - ConRFT 假设有个"能力大致够用"的 IL policy
  - HIL-SERL 更通用,但收敛慢
- 我们的选择:**HIL-SERL 做 RL 侧探索,ConRFT 思路作为 IL 模型的后续微调手段**

### Q4.4 "为什么不纯 RL?"

**核心答案**:
- 纯 RL 在机器人上**样本效率太低**,真机每天撞断几次就代价爆炸
- HIL / ConRFT 路线本质上是"**专家数据引导 + RL 修正尾部**",这是真机任务上最务实的方案
- 业界共识:真机 VLA 的未来大概率是 **预训练 VLA + HIL 后训练**

## 主题 5:LeRobot 数据格式

### Q5.1 "LeRobot 是什么?为什么用它?"

**核心答案**:
- **LeRobot** 是 HuggingFace 开源的机器人学习框架,提供**标准化的数据集格式**(parquet + jsonl + mp4)和**统一的训练/推理接口**
- 优势:
  - 一套格式可以对接 ACT / DP / OpenVLA / π0 等多种模型
  - HuggingFace Hub 原生支持(可以 pull 公开数据集)
  - v3.0 开始支持 **Multi-LerobotDataset** 并行加载(我们训 4 类箱体混训时用到)
- 我们项目把 Rosbag 数据 → LeRobot v3.0 格式 → 对接所有模型训练

### Q5.2 "LeRobot v3.0 和之前版本区别?"

**核心要点**(了解范围):
- v3.0 支持 Multi-Dataset 并行加载,不需要为多任务混训重新转数
- 改进了 meta 信息和 episode 索引
- 我们实测:从 v0.3 迁到 v0.4.3 过程中发现 task_text embedding 会拖慢训练,后来改成**离线 BERT 编码**

### Q5.3 "Rosbag 怎么转 LeRobot?"

**核心要点**:
- 核心步骤:
  1. Rosbag 按 topic 对齐时间戳(相机、关节、夹爪)
  2. 按 arm_traj 等关键话题**自动裁剪无效段**(苏州 184 条 bag 这么做过)
  3. 图像按 episode 写 mp4,状态 / 动作写 parquet
  4. meta.jsonl 描述 episode 长度、任务
- 我们踩的坑:DP3 bag → zarr 转换,2.8GB bag 需要 **10min+**,早期没优化数据 IO

## 主题 6:Sim2Real Gap

### Q6.1 "你们怎么处理 Sim2Real Gap?"

**核心答案**(三层递进):
1. **仿真端尽量贴近实机**:
   - MuJoCo 物理仿真 + IsaacLab 渲染
   - **3D-GS + MASt3R-SLAM** 重建训练场 → 生成贴近真实背景的仿真数据
   - Domain randomization(箱子位置 / 朝向 / 大小 / 光照)
2. **仿真→实机 zero-shot 验证 gap**:
   - 杨小康在仿真训 ACT,先 zero-shot 看实机掉几成
3. **实机微调收尾**:
   - 真机采少量数据 finetune → 显著补齐 gap
4. **长期方案**:**数据增强(Cosmos-transfer)补训练分布**让模型见更多场景

### Q6.2 "3D Gaussian Splatting 在你们项目里怎么用?"

**核心答案**(了解范围):
- 用途:训练场场景重建 → 生成**任意视角**的仿真数据
- 流程:MASt3R-SLAM 恢复高精点云 → Poisson 重建 mesh → MeshLab 纹理恢复 → IsaacLab 导入 USD
- 坑:3D-GS 在无特征的地面 / 墙面会出现"气泡",丢失重建细节(0509 周报)

**坦诚话术**:
> "3D-GS 这块主要是杨小康在做,我参与的是下游的数据生成 pipeline。具体的 rasterization / 可微渲染细节我不深入。"

## 主题 7:数据增强(我主导) ★ 面试重点

### Q7.1 "你主导的数据增强具体做了什么?"

**核心答案**(你的亮点):
- 背景:Groot VLA 在 in-dist 做到 85%,但杂乱背景 OOD 降到 83%,**倾斜摆放降到 0%**。说明训练数据分布窄,模型见不到的场景就崩
- 我的方案:
  1. **主方案:Cosmos-transfer**(NVIDIA 2025 发布的可控视频生成模型)
     - 输入:原始真机视频 + 文本描述(如"夜晚仓库、金属地面")
     - 输出:同样动作但背景/光线完全替换的合成视频
     - 优势:保留动作轨迹一致性,只改变视觉分布
  2. **备选方案:SAM3+IC-light**
     - SAM3 分割前景(机器人 + 箱子)→ IC-light 对背景重新光照渲染
     - 优势:更轻量、更可控;可与 Cosmos-transfer 组合(Cosmos 做大幅背景,IC-light 做光线微调)
- 目标:把 OOD 杂乱背景 83% → 90%+,倾斜 0% 往上拉

### Q7.2 "为什么选 Cosmos-transfer 而不是传统 GAN 增强?"

**核心要点**:
- 传统 GAN 增强(CycleGAN 等)**动作一致性差**:背景换了但可能连机器人姿态也扭了
- Cosmos-transfer 基于视频扩散模型,**时序一致性好**,不会动作变形
- 另外:Cosmos 是 NVIDIA 针对机器人 / 物理仿真专门训练的,先验知识更贴合我们场景
- 缺点:生成质量不稳定,有时出现物体消失 / 形变 → 所以要配质量筛选

### Q7.3 "DINO+SAM+LaMa 怎么自动擦 TAG?"

**核心答案**(你参与):
- 背景:真机采集时箱子贴 AprilTag 做视觉伺服,模型会过拟合到 TAG 特征。上线场景没有 TAG,就崩
- 流水线:
  1. **DINO** 检测 TAG 的 bounding box(开放词汇检测,文本提示 "tag" / "marker")
  2. **SAM** 以 bbox 为 prompt 生成精确 mask
  3. **LaMa** 以 mask 为区域做图像修复(inpainting)→ 把 TAG 区域 "变回" 箱子纹理
- 性能:单相机 parquet 转一份 8 分钟,4/5 层 4 相机全数据集 **6 小时跑完**
- 效果:模型在去 TAG 数据上 loss 正常下降,推理稳定性改善

### Q7.4 "你们有没有量化数据增强的收益?"

**坦诚话术**:
> "1219 这一周我们才刚启动 Cosmos-transfer 这条线,**量化对比还没做完**。目标是 AB 测试:Groot 用原数据 vs 用原数据 + Cosmos 合成数据,看 OOD 杂乱背景和倾斜摆放的成功率增益。这个实验会在下一阶段完成。"

这样比编造一个数字诚实得多,**面试官会欣赏**。

### Q7.5 "数据增强的失败模式?"

**核心要点**:
- **动作-视觉不一致**:Cosmos-transfer 有时候生成视频里箱子位置和原动作对不上 → 需要 action space 质量筛选
- **过度增强**:背景换得太离谱,反而是"假分布"
- **对抗示例**:合成数据可能让模型学到"泛化的错误假设"
- 应对:**合成数据占比控制**(真机 > 合成),保留原真机 baseline

## 主题 8:MPC / WBC / IK(控制层)

> 这是你不主导的模块,**了解 + 诚实** 是最好的策略。PDF 《MPC+WBC算法中,包含了哪些核心公式?》是你的底牌。

### Q8.1 "KMPC 是什么?"

**核心答案**(低风险版本):
- KMPC = **运动学 MPC**(Kinematic Model Predictive Control),不涉及动力学,只做末端/质心轨迹规划
- 我们用的是 **OCS2 框架**
- 输入:当前状态 + 目标轨迹;输出:未来 N 步的关节 / 末端指令
- 踩的坑:KMPC 延迟 200+ms(0606/0614 根因还没定位);yaw 角不归一化累加到 100rad 会求解崩溃(0726)

**面试话术**:
> "KMPC 的具体求解细节(QP 结构、任务优先级)是控制组游桦、吴泽锐在做。我作为 VLA 算法侧,理解 KMPC 作为 action space 的下游 → 我的 ACT 或 Groot 输出 end-effector 位姿,KMPC 负责求解到关节。"

### Q8.2 "WBC 和 KMPC 什么关系?"

**核心要点**:
- WBC = Whole-Body Control,**全身多任务优先级控制**,通常是 QP 求解
- 关系:KMPC 做轨迹规划,WBC 做瞬时全身控制;**KMPC 输出参考 → WBC 跟踪**
- 我们有 **DirectWBC 模式**,不走 KMPC,直接发关节指令(用于 comACT 等需要低延迟的场景)

**参考 PDF** 可能考到的公式:
- 浮动基座动力学方程:`M(q)q̈ + h(q,q̇) = S^T τ + J^T λ`
- WBC 任务优先级 QP:`min ||J_i q̈ - ẍ_i||²` s.t. 上层任务约束
- KKT 条件下的 QP 求解

### Q8.3 "IK 有几种方案,怎么选?"

**核心答案**:
- 我们对比过三种:
  1. **一阶段数值 IK**:最快,精度一般
  2. **两阶段数值 IK**:先粗解再精调,精度好但慢
  3. **解析解 IK**(四书 closed-form):最快且精确,但需要特定机械构型
- 选型:Kuavo 双臂用**解析解**(精度 + 速度都好);轮臂 MT-7 用**两阶段数值**(构型复杂)
- 工程上还要加 **IK 跳变保护**(相邻帧解的距离阈值,防止分支跳变)

### Q8.4 "action space 怎么选?cmd_vel / cmd_pose / joint position?"

**核心答案**(踩坑最深的一题):
- 我们试过 **cmd_vel**(速度指令)、**cmd_pose**(位姿指令)、**cmd_pose_world**(世界系位姿)
- 踩的坑:
  - cmd_pose **时间轴不连续**,ACT 学不会"停下来",base 总是冲向箱子
  - cmd_pose_world 过小的点被过滤
- 最终:**走路用 cmd_vel,手臂精操作用 cmd_pose**
- 踩坑过的 delta joint position:手臂关节逐渐偏移(累积误差)
- **结论**:action space 的选择不是"哪个好",而是"任务阶段需要什么",走路和抓取可以用不同 action space

## 主题 9:推理加速 / 评测体系

### Q9.1 "ONNX 怎么转 TensorRT?加速比多少?"

**核心答案**:
- 流程:`.safetensors → torch.onnx.export → ONNX → trtexec 转 .engine`
- 性能:
  - ONNX 首次推理 **4s 预热**(层图构建 + 权重加载)
  - ONNX 后续推理 **0.1s / 帧**
  - **TensorRT 30Hz**(约 33ms / 帧),相比 ONNX 提升 3 倍
- 关键:FP16 / INT8 量化 + 图优化 + kernel fusion
- 实测 Orin Jetpack 5.1.3 环境下稳定

### Q9.2 "推理加速会不会损失精度?"

**核心答案**:
- 理论上 FP16 会有微小精度损失,但实测 Groot 在 Orin 上精度无明显下降
- 更大风险是:
  - **skip-chunk-ratio**(跳过部分 chunk 推理以加速,从 11s → 9s)这个会损失准度 → 我们在成功率和速度之间做 trade-off
  - ONNX 导出时某些 op(比如动态 shape)可能 fallback 到 CPU,性能暴跌

### Q9.3 "评测体系怎么设计?"

**核心答案**:
- 分三层:
  1. **In-distribution**:训练场景完全一致(白底、标准位置) → Groot 85%
  2. **OOD 轻度**:相似场景但变量变化(杂乱背景 83%、金属重物 100%)
  3. **OOD 重度**:corner case(倾斜摆放 0%)
- 每层都采独立小规模评测集(20-30 episode),手工标注成功 / 失败
- **关键:评测集绝对不能和训练集污染**,这是校招算法岗的基础常识

### Q9.4 "你们的失败案例怎么分析?"

**核心答案**(Dagger 闭环):
- 每次失败 episode 记录:现象 + 哪一步断掉 + 图像/状态回放
- 分类:
  - **感知失败**(箱子定位不准)
  - **规划失败**(走到位置但手不对)
  - **控制失败**(IK 跳变、KMPC 抖动)
- **Dagger 补采**:针对某类失败定向采数 → 重训 → 再评测
- 这个循环是**杨小康主导**的框架,我参与数据侧

### Q9.5 "最后 20% 的成功率怎么提?"

**核心答案**(面试官最爱问):
- 前 80% 靠数据量,**最后 20% 靠数据质量 + 边界 case 补采 + OOD 增强**
- 我们的规划:
  1. **Cosmos-transfer 合成数据** 扩展 OOD 分布(我主导)
  2. Dagger 针对倾斜 / 遮挡等 corner case 补采
  3. 模型路线:Groot → 可能的 π0.5 或更大 VLA
  4. 多模态增强:加 3D 语义(DP3 / PointCloud 编码)
