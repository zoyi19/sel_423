## 主题 1:VLA 架构 —— OpenVLA / RDT / π0 / π0.5 / Groot 的差异

### Q1.1 "你们尝试过哪些 VLA 模型,为什么最后选 Groot?"

**核心答案框架**:
1. 团队做过 5 类 VLA 对标:OpenVLA-OFT、RDT-1B、π0-fast、**π0.5**、**Groot**
2. 选型维度:数据需求、推理速度、多任务能力、实机部署友好度
3. **Groot(SigLIP + 全参微调)**最终胜出:
   - 多任务混训能力强(4 类箱体混训 85%),OpenVLA-OFT 单任务 98.8% 但多任务混
   - 推理可经 ONNX → TensorRT 压到 30Hz,RDT/π0-fast 推理慢
   - 数据规模 2900+ 条真机够用
4. **π0.5 也试了**(H100 10h 跑 3K step),但在我们的 LeRobot v3.0 管线上需要更长训练周期
5. 踩过的坑:RDT-1B / π0-fast 拿 50 条数据训,只学走未学抬手,说明大 VLA 对数据规模敏感

### Q1.2 "Groot 架构核心是什么?"

**核心要点**(讲你把握得住的):
- Groot(NVIDIA GR00T N1 / N1.5):**双系统架构**(快系统 + 慢系统)
- 视觉:SigLIP;action:Diffusion Transformer(DiT)head 做连续动作生成
- 差异核心:
  - OpenVLA 离散 action token,Groot 连续 action via DiT
  - Groot 双系统更适合长视地(慢系统规划,快系统执行)

### Q1.3 "π0 / π0.5 区别?"

**核心要点**:
- π0:PaliGemma-3B + flow matching action head
- π0.5:更大规模预训练 + 跨机器人迁移能力 + 更强指令理解
- 我们实测 π0.5:LeRobot v3.0 格式下 H100 10h / 3K step,**未达可用水平**(数据不够 / 训练周期不够)

**坦诚话术**:
> "π0.5 的微调主要是顾家伟在推,我参与的是数据格式对接(LeRobot v3.0)和前期配置。具体 pre-train 细节我没深入。"

## 主题 2:ACT 原理与 chunk size

### Q2.1 "讲一下 ACT 的核心设计?"

**核心答案**:
- **ACT = Action Chunking Transformer**(Aloha 论文)
- 关键设计:
  1. **Action Chunking**:预测未来 K 步动作(K 通常 16-100)
  2. **Temporal Ensemble**:每步执行时对过去多次推理结果做加权平均
  3. **CVAE 结构**:encoder 接观测 + 动作 → latent z → decoder 生成动作
- 团队用 ACT 做到单摞 80% 成功率(**我入职前达成**),这是我后面 pytrees 封装要对接的策略之一

### Q2.2 "chunk size 怎么选?"

**核心要点**:
- **太大**(我们试过 100)→ 覆盖任务阶段切换点,模型学不准;过拟合训练轨迹
- **太小**(比如 8)→ ensemble "预测视野"小,平滑度差
- 我们踩的坑:chunk_size=100 搬箱反而比 32 差
- **搬箱最终用 chunk_size=32 到 64**

**我可以讲的消融经验**(阶段 2 我辅助组里跑过):
> "拆垛阶段我做过 chunk size 的消融辅助,在 Groot 上对比 16 / 24 / 32 的成功率和推理延迟。结论是 24 是我们场景的甜点,16 推理快但成功率掉 5% 左右,32 的平滑度改善有限反而推理慢。"

### Q2.3 "chunk 边界跳变怎么处理?"

**核心答案**:推理平滑三件套
1. **Inpainting**:chunk 重叠部分前尾+当前头加权融合
2. **低通滤波**:对 delta action 用 Savitzky-Golay 滤波(Δt=0.25s)
3. **插值**:chunk 内对 target 线性插值
- 这套在 comACT 上消除实物抖动(1114 周报,杨小康主导,我参与了 pytrees 节点里的推理平滑接口)

## 主题 3:Diffusion Policy / DP3

### Q3.1 "DP 和 DP3 区别?"

**核心要点**:
- **DP**:2D 图像 + CNN encoder + DDPM denoise 预测动作
- **DP3**:输入改成 **3D 点云 + PointNet encoder**
- 优势:DP3 对视角 / 光照鲁棒性更好,推理最快
- 劣势:我们实测 DP3 **方向理解有误差**,3D 语义还不够强;噪声鲁棒性:训练噪声 0.02,测试 0.0/0.01/0.02 成功率 0.42/0.34/0.32

### Q3.2 "DP 的 horizon 和 n_obs_step?"

- **horizon**:预测未来多少步,常用 8
- **n_obs_step**:输入几帧历史,n_obs_step=2 推理启动快但视觉歧义多,=17 启动慢但歧义少
- Trade-off:**推理速度 vs 启动延迟 vs 视觉歧义**

### Q3.3 "DP 推理能力突然消失?"

**核心答案**(团队踩坑):
- 现象:某些初始位置推理可成功,推理多次后能力消失
- 根因:image encoder 对相机序号敏感 + OOD 分布
- 解法:Dagger 补采边界数据 + finetune encoder
- 这个 DP OOD 问题是我后来 Groot DAgger 失败恢复借鉴的前车之鉴

## 主题 4:LeRobot 数据格式

### Q4.1 "LeRobot 是什么?"

**核心答案**:
- HuggingFace 开源机器人框架,**标准化数据集格式**(parquet + jsonl + mp4)
- 优势:一套格式对接 ACT / DP / OpenVLA / π0 多种模型;v3.0 支持 **Multi-LerobotDataset 并行加载**
- 项目里:**Rosbag 数据 → LeRobot v3.0 格式 → 对接所有模型**,这套管线我在阶段 1 参与搭建维护

### Q4.2 "v3.0 和 v2 区别?"

**核心要点**:
- v3.0 支持 Multi-Dataset 并行加载,不需要多任务混训重新转数
- 改进了 meta 信息和 episode 索引
- 我们从 v0.3 迁到 v0.4.3 过程中发现 task_text embedding 拖慢训练,改为**离线 BERT 编码**

### Q4.3 "Rosbag 怎么转 LeRobot?"

- 核心步骤:
  1. Rosbag 按 topic 时间戳对齐(相机 / 关节 / 夹爪)
  2. 按 arm_traj 关键话题**自动裁剪无效段**(184 条苏州 bag)
  3. 图像按 episode 写 mp4,状态/动作写 parquet
  4. meta.jsonl 描述 episode 信息
- 踩的坑:DP3 bag → zarr,2.8GB bag 需 10min+(早期未优化 IO)
- **这套管线的维护是我在阶段 1 的工作之一**

### Q4.4 "你具体写的 bag → LeRobot 转换器做了哪些事?"★ 硬核追问

**核心回答**(这是我亲手写的 ~1400 行脚本):

1. **多 topic 时间戳对齐 + Forward Fill**:
   - dt=0.1s(10Hz)构造均匀时间轴
   - Forward fill 策略有讲究:第一帧用**第一个有效消息真实值**(避免 delta 冷启动大跳变),后续无消息帧**填零**,有历史就正常前向填充

2. **混合 Action Space 设计**(action 24 维):
   - **COM(质心)9 维 = 3 位置 + 6D 旋转,全部 delta**(相对增量)
   - **Arm(双臂)14 维 = 7+7 关节,absolute 弧度**
   - **Gait(步态)1 维 = 1(walk) / 2(stance),discrete**
   - 理由:走路用 delta 可泛化,关节必须 absolute 避免累积误差,步态是分类变量

3. **6D 旋转表示**(Zhou et al. 2019):
   - 不用欧拉角(万向锁+不连续),不用四元数(double cover 对回归不友好)
   - 取旋转矩阵前两列 Fortran order 展开
   - 重构时 Gram-Schmidt 正交化 + 1e-8 epsilon 保护

4. **Previous Delta 放进 State**(关键 trick):
   - observation.state = **38 维** = `[14 arm + 3 lin_acc + 3 ang_vel + 9 com_6d + 9 prev_delta]`
   - 让模型看到"上一帧走了多少",消除 chunk 边界不连续

5. **积分面积保持的平滑**:
   - Savitzky-Golay(win=11, poly=3)平滑 delta 序列
   - 平滑会改变总位移 → 计算 scale factor 缩放,保持积分面积不变
   - 验证每维相对误差 < 1%

6. **图像特征预提取**(对应 1018 周报 80 条 2h 跑 250 epoch):
   - ResNet18(ImageNet pretrained)+ FrozenBatchNorm2d
   - 取 layer4 feature map,240×320 分辨率
   - 特征 hex 编码存 binary bytes,训练时直接加载不过 encoder

**可能的追问与回答**:

- **"为什么 COM 用 delta 但 Arm 用 absolute?"** → 走路是连续长距离,delta 学位移率更易泛化;关节必须精准到绝对角度,delta 累积误差会让抓取偏
- **"6D 旋转表示具体公式?"** → 旋转矩阵 R 的前两列 [R:,0, R:,1],Fortran order 展开成 6 维;重构时第一列归一化 → 第二列减去第一列投影后归一化 → 第三列用叉积
- **"积分面积为什么要保持?"** → 平滑的动机是让轨迹漂亮,但平滑会改变"总位移",机器人走错位置。scale factor 调整后,最终位置不变,只是过程更柔和
- **"Forward fill 为什么第一帧特殊处理?"** → 如果第一帧填零,第二帧是真实位姿,delta 会瞬间巨大,模型学坏;用"第一个有效消息"做 warm start
- **"arm 数据为什么要转弧度?"** → ROS `/mm_kuavo_arm_traj` 话题原生是度,LeRobot 统一用弧度,否则训出来的模型 action 偏差 57 倍

## 主题 6:数据增强(**我主导,SAM3+IC-light 是最终落地,Cosmos-transfer 是对比后放弃**)★ 面试重点

### Q6.1 "你主导的数据增强具体做了什么?"

**核心答案**(你的亮点):
- **背景**:Groot 在 in-dist 85%,但杂乱背景 OOD 降到 83%,**倾斜摆放降到 0%**。数据分布窄是瓶颈
- **调研三条路线**(由浅到深逐个排除):
  1. **OpenCV 传统增强**(亮度 / 对比度 / 翻转 / 噪声):像素层操作,改不了场景结构和语义,真机 OOD 测试**几乎无提升,舍弃**
  2. **Cosmos-transfer**(NVIDIA 视频扩散):整段视频背景替换,时序一致性好。**但 4090 跑不动,需要 H100 + 单视频生成数分钟,公司算力不能支持全数据集**
  3. **SAM3 + IC-light**(选):SAM3 分前景,IC-light 重光照。更轻量可控
- **我的评估决策**:**OpenCV 太浅、Cosmos 算力不够,落地 SAM3+IC-light**
- 这个决策本身是一个工程亮点 —— 展现你会**权衡理论最优和工程可行**

### Q6.2 "为什么选 SAM3+IC-light 不选 Cosmos-transfer?"

**核心要点**:
- Cosmos-transfer 是视频扩散,**推理时间 10s+/segment**,全数据集 2900+ 会跑几天
- SAM3 + IC-light 是 **image-level 处理**,**单 4090 可跑**,快得多
- **量化对比**:**单个 3 分钟操作视频,SAM3+IC-Light 在 ~10 分钟内可生成 12 个不同背景的视频**
- Cosmos-transfer 质量波动:有时箱子消失 / 形变 → **动作-视觉不一致**,不能直接用
- SAM3 的 mask + IC-light 重光照**只改光照和颜色**,保留所有动作相关几何 → 更安全

### Q6.3 "SAM3+IC-light 具体流程?"

**核心要点**:
1. **SAM3 分割**:对每帧图像,检测机器人 + 箱子的前景 mask(开放词汇 prompt)
2. **背景提取**:mask 的反向区域作为可修改背景
3. **IC-light 重光照**:用不同光照条件(例如"夜晚仓库"、"强逆光"、"荧光灯")重新渲染背景
4. **合成回原图**:前景保持不变,背景换成新渲染的
5. **批量处理**:对 2900+ 条真机数据做 3-5 种光照变体,数据量扩张到 1 万条量级

### Q6.4 "Cosmos-transfer 和传统 GAN 数据增强比?"

**你可以说明的**:
- 传统 GAN(CycleGAN 等)**动作一致性差** —— 背景换了但可能连机器人姿态也扭了
- Cosmos-transfer 基于视频扩散,**时序一致性好**,理论上应该是最好的
- **但工程成本高** → 回到 SAM3+IC-light 这种**务实的 image-level 方案**
- 教训:**最优理论方案不一定是最适合工程落地的方案**

### Q6.5 "量化效果怎么样?"

> "AB 测试因离职时间没跑完,**有初步信号 25% 合成数据让 OOD 杂乱背景有提升**,完整对比留给后续。**不要编造数字**。"

### Q6.6 "DINO+SAM+LaMa 擦 TAG 和你这个 SAM3+IC-light 什么关系?"

**核心要点**:
- DINO+SAM+LaMa 是**早于我**的工作(周贤涵主导,1024 周报),我入职时刚在收尾
- 目的不同:
  - DINO+SAM+LaMa = **擦除不想要的东西**(AprilTag 干扰特征)
  - SAM3+IC-light = **增加想要的多样性**(背景光线变化)
- **我做的是在这个数据流水线基础上扩展** —— 不是替代,是互补

### Q6.7 "OpenCV 传统增强为什么不行?"

**核心要点**:
- OpenCV 提供的是**像素层增强**:亮度 / 对比度 / 翻转 / 裁剪 / 高斯噪声 / 颜色抖动
- 问题:**改不了场景结构和语义**
  - 改背景颜色 ≠ 改成"杂乱仓库背景"
  - 改对比度 ≠ 改成"夜晚 / 强逆光"
- 真机 OOD 测试:**几乎无提升**(因为模型见过的是同一个场景的颜色变化,不是新场景)
- 启发:**数据增强的有效性取决于"扩展的是什么层"**
  - OpenCV → 像素层(浅)
  - SAM3+IC-Light → 光照 + 颜色语义层(中)
  - Cosmos-transfer → 场景结构 + 视角层(深,但贵)
- 我们最后选 SAM3+IC-Light 是**在"扩展深度"和"工程成本"之间的平衡点**

### Q6.8 "Cosmos 在更好显卡下值得重新评估吗?"(钩子追问预案)

**核心答案**:
- 值得。**SAM3+IC-light 改不了场景结构和视角,Cosmos 能改**
- 如果有 8×H100,我会做 3 组对比:
  - 组 1:原数据 + SAM3+IC-Light(我做过)
  - 组 2:原数据 + Cosmos-transfer
  - 组 3:原数据 + 两者组合(SAM3 改光照 + Cosmos 改场景结构)
- 我猜组 3 最好,但需要算力验证

## 主题 7:MPC / WBC / IK(控制层)

> 不是你主导的模块,**了解 + 诚实** 是最好策略。PDF《MPC+WBC算法中,包含了哪些核心公式?》是底牌。

### Q7.1 "KMPC 是什么?"

**核心答案**:
- KMPC = **运动学 MPC**,不涉及动力学,只做末端 / 质心轨迹规划
- 用 **OCS2 框架**
- 踩的坑:延迟 200+ms;yaw 角累加到 100rad 求解崩溃(加了上限约束)

**话术**:
> "KMPC 具体求解(QP 结构、任务优先级)是控制组游桦、吴泽锐在做。我理解 KMPC 作为 action space 的下游 —— ACT 或 Groot 输出 end-effector 位姿,KMPC 求解到关节。"

### Q7.2 "WBC 和 KMPC 关系?"

- WBC = Whole-Body Control,**全身多任务优先级**,通常 QP 求解
- 关系:**KMPC 做规划,WBC 做瞬时跟踪**
- DirectWBC 模式:不走 KMPC,直接发关节指令(用于 comACT 等低延迟场景)

**PDF 可能考公式**:
- 浮动基座动力学:`M(q)q̈ + h(q,q̇) = S^T τ + J^T λ`
- WBC 任务优先级 QP:`min ||J_i q̈ - ẍ_i||²` s.t. 上层任务约束
- KKT 条件

### Q7.3 "IK 方案选型?"

- 三种方案:一阶段数值 / 两阶段数值 / **解析解**(四书 closed-form)
- 双臂 Kuavo 用解析解(精度 + 速度好);轮臂 MT-7 用两阶段数值
- IK 跳变保护:相邻帧解距离阈值,防分支跳变

### Q7.4 "Action space 踩过什么坑?"

- 试过 cmd_vel / cmd_pose / cmd_pose_world
- 踩的坑:**cmd_pose 时间轴不连续**,ACT 学不会停下来;**cmd_pose_world 过小点被过滤**
- **最终结论**:走路用 cmd_vel,手臂精操作用 cmd_pose

## 主题 8:推理加速 / 评测体系

### Q8.1 "ONNX 转 TensorRT 加速比?"

- `.safetensors → torch.onnx.export → ONNX → trtexec`
- 性能:
  - ONNX 首次 **4s 预热**,后续 0.1s/帧
  - **TensorRT 30Hz**(33ms/帧),比 ONNX 提升 3 倍
- 关键:FP16 / INT8 + 图优化 + kernel fusion
- **我在阶段 1 的 pytrees 推理节点封装时对接过这套加速链路**,知道 engine 文件加载 / 输入输出绑定这些细节

### Q8.2 "加速会不会损失精度?"

- FP16 实测 Groot 在 Orin 上无明显下降
- 更大风险:**skip-chunk-ratio**(跳部分 chunk 加速 11s→9s)会损失准度 → 在成功率和速度间 trade-off

### Q8.3 "评测体系?"

**核心**:分三层
1. In-distribution:训练场景一致(白底) → 85%
2. OOD 轻度:相似但变量变化(杂乱 83%、重物 100%)
3. OOD 重度:corner case(倾斜 0%)
- 每层独立小评测集(20-30 episode),手工标注成功/失败
- 视觉来源:3 摄(头部 + 双腕),评测集覆盖 3 个视角的一致性
- **关键:评测集绝对不能和训练集污染**

### Q8.4 "失败案例怎么分析?"

**DAgger 闭环**(见主题 11):
- 每次失败记录:现象 + 哪一步断掉 + 图像/状态回放
- 分类:感知失败 / 规划失败 / 控制失败
- 定向补采 → 重训 → 再评测

### Q8.5 "最后 20% 怎么提?"

- 前 80% 靠数据量,**最后 20% 靠数据质量 + 边界 case 补采 + OOD 增强**
- 我们规划:
  1. **SAM3+IC-light 扩展 OOD 分布**(我主导)
  2. **DAgger 针对倾斜 / 遮挡补采**(我主导)
  3. 模型:Groot → 可能的 π0.5 或更大 VLA
  4. 多模态:加 3D 语义(DP3 / PointCloud)

## 主题 9:UMI 无本体采集(**我主导,阶段 3**)★ 面试最大亮点

### Q9.1 "UMI 是什么?"

**核心答案**:
- UMI = Universal Manipulation Interface,Cheng Chi et al. RSS 2024
- 核心思想:**手持夹爪 + 相机,采集人类演示数据,无需机器人本体**
- 原理:
  1. 手持夹爪(类似 iPhone grip)+ 视觉相机
  2. 记录相机位姿(SLAM / 动捕)+ 夹爪开合状态
  3. 转换为机器人 end-effector 动作
- 优势:采数成本极低,任何人都能采
- 挑战:**embodiment gap** —— 无本体数据训出的模型迁到机器人会掉点

### Q9.2 "你们 UMI 怎么落地的?"

**核心答案**:
1. **方案构建**:参考论文搭建硬件和软件 pipeline
2. **关键选择 1:放弃 ORB-SLAM → 动捕**
   - ORB-SLAM 在训练场跟踪失败(无特征墙面 / 快速运动)
   - 改用 Vicon/OptiTrack 获取毫米级位姿
   - Trade-off:精度 vs 场地限制
3. **关键选择 2:本体实时映射**
   - UMI 坐标系 → 机器人 base → end-effector
   - 速度 / 加速度平滑(UMI 采集速度可能超过机器人关节极限)
4. **关键选择 3:UMI vs VR 对比** —— 各有利弊(见主题 9.3)

### Q9.3 "UMI / VR / 真机遥操 三方怎么对比?"

| 维度 | 真机遥操(基线) | VR(Pico/Quest3/MetaVR) | UMI(动捕) |
|---|---|---|---|
| 可操作性 | 中 | **过于沉重,1-2h 疲劳** | **方便,长期不累** |
| 场地 | 不限 | 不限 | **受动捕范围限制** |
| 数据质量 | **最好(直接训机器人)** | **传输不均匀,数据不平衡,不能直接训练** | 动作连续,位姿精准(动捕级) |
| 采数效率 | 低 | 低(专业操作员) | **高(任何人)** |
| embodiment gap | **0** | 小(VR 直接驱动真机) | **大(精细任务尤甚)** |
| 适用任务 | 全部 | 演示 / 小规模 | **粗精度大量数据** |

**讲述要点**:
- VR 是**首先评估并放弃**的方案 —— 设备沉重 + 数据不平衡 + 不能直接训练
- UMI 是**最终选定**的预研方向 —— 采数效率最高,但有 embodiment gap
- 真机遥操是**对照基线** —— 不是预研要替代的目标,数据始终最好
- **三种方案不是互相替代,是互补**:精细任务用真机,粗任务可以 UMI

### Q9.4 "为什么放弃 ORB-SLAM?"

**核心答案**:
- 训练场墙面地面**特征点少**(白墙、无纹理地板),SLAM 特征点丢失
- 手持夹爪运动**速度快 + 姿态变化大**,ORB-SLAM 跟踪失败率高
- 结果:**有些 trajectory 中途漂移,或者直接 lost tracking**
- 替代方案动捕:精度高但场地受限 → 我们选动捕,场地限制在我们场景可接受

### Q9.5 "最后得出什么结论?"

**核心答案(最有分量的一句)**:
> "方案由我主导设计,和一位同事合作采集了 **1000 条'放置小零件到盒子内部'** 任务数据,用 **Lingbot-VLA**(开源 VLA,与公司有合作关系)训练,**成功率只有 30% 出头**。这是一个明确的**负面结论** —— 无本体采集在精细操作任务上存在显著 embodiment gap,效果不及预期。我基于这个结果给团队的建议是:**UMI 只投入粗精度任务(拾取 / 搬运),精细操作仍用真机 VR 遥操**。"

### Q9.6 "为什么是 embodiment gap?"

三层(详见 06 §5.4):**视角**(UMI 夹爪 vs 机器人头/腕)+ **夹爪形态**(UMI 塑料 vs 机器人金属)+ **精细任务放大 gap**(毫米级精度被任何差异放大)。粗任务影响小,精细任务致命。

### Q9.7 "Lingbot-VLA 是什么?"

**标准答案**:
> "Lingbot-VLA 是一款**开源 VLA 模型**,我们公司和它的团队有合作关系,所以我们在实验中拿它作为无本体数据质量验证的基线模型。选它的原因是:**开源可复现**、**有公司合作确保配置对齐**、以及**在类似精细操作任务上有公开 benchmark 可对标**。"

**延伸追问预判**:
- "为什么不用自研 VLA?" → 因为自研 VLA(GR00T / π0.5)都是基于真机数据训练的,直接拿来训无本体数据会引入模型和数据源两个变量,不能单独验证 UMI 数据的效果。Lingbot-VLA 作为中立基线,更能隔离出"数据源"这个单一变量。
- "Lingbot-VLA 的 recipe 你清楚吗?" → 公开文档里的 training recipe 我复现过(数据量 / 学习率 / chunk size 等),有公司合作渠道时也可以对齐细节,实验结果是可控的。

### Q9.8 "如果继续做,你会怎么改?"

**展现思考深度**:
> "三个方向:
>
> 1. **UMI + 真机 co-training**:用少量真机'锚定',大量 UMI'扩展',可能是更现实的路径
> 2. **UMI 数据只用于粗精度任务**:放弃精细任务的 UMI 采集,专注拾取 / 搬运
> 3. **动捕 → 纯视觉位姿估计**:长期看动捕场地限制是瓶颈,可以用 FoundationPose 或 3DGS + 视觉 SLAM 的组合,摆脱动捕"

## 主题 10:pytrees 行为树 + 推理节点编排(**我主导,阶段 1**)

### Q10.1 "为什么用行为树,不直接写 state machine?"

**核心答案**:
- 机器人任务有**子任务复用** + **失败回退** + **并行监控**需求
- **行为树(Behavior Tree)**天然支持:
  - 组合节点(Sequence / Selector / Parallel)
  - 失败上抛机制(一个子任务失败,父节点决定切换到其他子树)
  - 条件节点 / 动作节点分离
- 相比 state machine:**行为树更易扩展和维护**,尤其在多策略切换场景

### Q10.2 "py_trees 具体怎么用?"

**核心答案**:
- **py_trees** 是 Python 实现的行为树库
- 基本元素:
  - `Behaviour` 基类(tick → SUCCESS / FAILURE / RUNNING)
  - 组合节点:`Sequence`(全部成功才成功)/ `Selector`(一个成功就成功)
  - Blackboard(节点间共享数据)
- 我做的:**把 VLA 模型封装成 Behaviour 子类**

### Q10.3 "你封装的 pytrees 推理节点什么样?"

**核心思想(一句话)**:
> "把**接入一个新策略**这件事变成**按标准格式实现一个 Adapter** —— 上层行为树只依赖两个标准 schema(`ObservationFrame` / `ActionChunk`),任何新模型写一个 Adapter 对齐到这两个 schema 即可,上层代码零改动。模板方法 + Adapter 模式。"

**为什么需要这层封装 —— 四个策略的入口差异**:

| 策略 | 输入 | 输出 | 特殊点 |
|---|---|---|---|
| ACT | multi-cam + state(单帧) | chunk (32, action_dim) | 单帧输入,一次出 chunk |
| DP / DP3 | n_obs_step 历史帧 + state | horizon 步 | 维护 obs queue,DDPM 多步 denoise |
| flow_matching | 同 DP | **multi-head**:arm + gripper | 推理 20Hz,多头要分开解码 |
| Groot / π0.5 | multi-cam + state + **language** | DiT 连续 chunk | 吃 TensorRT engine,首次加载 4s |

**具体差异**:
1. 观测拼装:ACT 当前帧 vs DP 历史帧窗口 vs Groot 还要语言指令
2. 推理调用签名:`select_action(obs)` / `predict_action(obs_dict)` / `forward(obs, text, state)` / TRT bind I/O
3. action space:joint_pos / eef_pose / cmd_vel / cmd_pose 混杂,多头模型夹爪单独一路
4. 发布频率:<10Hz / 20Hz / 30Hz 不同
5. 加载耗时:pth 秒级 vs TensorRT 4s 预热

**两个标准 schema**:

```python
@dataclass
class ObservationFrame:
    images: Dict[str, np.ndarray]   # {"head": ..., "left_wrist": ..., "right_wrist": ...}
    state:  np.ndarray              # joint pos + gripper
    task_text: Optional[str]        # Groot / π0.5 才用
    timestamp: float

@dataclass
class ActionChunk:
    actions: np.ndarray             # (chunk_size, action_dim)
    action_space: str               # "joint_pos" | "eef_pose" | "cmd_vel" | "cmd_pose"
    execute_freq: float             # publisher 按这个频率发布
    gripper: Optional[np.ndarray]   # 多头模型单独出的夹爪
    metadata: Dict                  # 置信度 / chunk 序号 / latency
```

**ModelInferenceNode(py_trees.Behaviour 子类)**:
- **状态机**:`IDLE → LOADING → INFERRING → PUBLISHING → DONE / FAILED`
- 持有一个 Adapter 实例(策略相关),只与两个 schema 交互
- `update()` 每次 tick 根据状态推进:加载(异步线程,tick 返 RUNNING)→ 读 blackboard 上的 `ObservationFrame` → Adapter 推理 → 发 `ActionChunk` 给 publisher → 执行完回 SUCCESS / 超时异常回 FAILURE

**关键设计**:
1. **策略接入零上层改动**:新模型进来,只写一个 Adapter 子类(~50 行),`ModelInferenceNode` 和行为树编排完全不动
2. **配置驱动切换**:YAML 改 adapter 类型 + 权重路径 → 换模型,不重启
3. **统一错误处理**:CUDA OOM / 话题缺失 / 推理超时在 Adapter 内部 catch,统一上抛 FAILED 让上层 Selector 切备用
4. **异步加载**:避免 TensorRT 4s 预热阻塞行为树 tick

**Adapter 子类分工(面试可举一两个例子)**:
- `ACTAdapter`:取单帧 + state → `select_action(obs)` → 返 joint_pos chunk
- `DPAdapter`:维护 `deque(maxlen=n_obs_step)` 的 obs queue → `predict_action(obs_dict)` → 返 eef_pose
- `FlowMatchingAdapter`:类似 DP,但输出两个 head(arm / gripper)分别填入 `ActionChunk`
- `GrootAdapter`:preprocess(SigLIP + language tokenization)→ TensorRT bind input buffers → enqueue → parse output → 返 joint_pos chunk

### Q10.4 "和行为树搬箱什么关系?"

**核心要点**:
- 团队更早的"行为树搬箱"(0606 / 0815 周报)是**原子技能编排**:走 → 定位 → 抓 → 提 → 搬 → 放
- 我的 pytrees 推理节点**让 VLA 策略可以作为一个 Behaviour 嵌入这种编排**
- 比如:"tag 视觉伺服走到箱前"(原子技能)+ "VLA 抓取"(我的节点)+ "行为树判断完成"

### Q10.5 "踩过什么坑?"

详见 `01_项目总览.md §1.2` 的"踩过的坑"段(TRT 4s 异步加载 / 多摄时间戳对齐 / CUDA OOM 兜底)。

## 主题 11:DAgger + 失败恢复(**我主导,阶段 2**)

### Q11.1 "DAgger 是什么?"

**核心答案**:
- **DAgger = Dataset Aggregation**(Ross et al. AISTATS 2011)
- 传统 BC 的问题:**distribution shift** —— 模型部署时会走到训练集没见过的状态
- DAgger 解法:
  1. 用当前 policy 跑评测(会失败)
  2. 在失败状态,让**专家(人)**给出正确动作
  3. 把这些"**在 policy 访问的状态上**"的专家动作加入训练集
  4. 重新训练
- 核心:**训练集的状态分布匹配 policy 访问的状态分布**

### Q11.2 "你的 DAgger 失败恢复怎么做的?"

**核心答案**(我主导):
1. **建立失败评测集**
   - 从 Groot 每次压测抽取失败 episode,按失败模式分类:感知(箱定位不准)/ 规划(到位手不对)/ 控制(IK 抖动)
2. **DAgger 干预回流**
   - 部署 Groot 到失败场景(倾斜摆放 / 白塑料 / 杂乱背景)
   - 人通过 **VR 遥操 / 键盘接管**失败 trajectory
   - 干预数据录制时**打专属标签**(intervention=True)
   - 进入下一轮训练,**权重可调**(原数据 70-90%,干预数据 10-30%)
3. **其他数据源扩充**
   - **仿真反例采集**:IsaacLab 中故意构造 corner case
   - **合成数据增强**:与 SAM3+IC-light 联动
   - **主动采样**:下次真机采数优先采已知失败模式的场景

### Q11.3 "你在干预数据 ratio 上怎么调?"

**核心答案**:
- 纯 BC:干预数据和正常数据 1:1
- 问题:干预数据密度大,但可能 **distribution shift**(只补 corner case,偏离原分布)
- 我的**混合缓冲策略**:
  - 原数据占 70-90%,干预数据占 10-30%
  - 避免过度 overfit 到 corner case,同时提升 OOD 成功率
- 需要实验调 ratio,这个是我后期的工作

### Q11.4 "为什么选 DAgger 不选 RL/PPO?"(关键对照,π0.6 是 RL 路线)

**核心答案**:
- **RL 路线**(类似 π0.6 的思路)是用强化学习直接 fine-tune VLA,让模型从 reward 信号学失败恢复
- **真机 RL 不现实**(详见 Q11.7 RL 科普),四个硬伤:
  1. **PPO 是 on-policy**:每次模型更新就要扔掉之前所有真机数据重新采,真机数据贵浪费不起
  2. **样本效率低**:PPO 论文需要 10^7-10^8 步,真机量级不可能
  3. **Reward 难设计**:拆垛任务怎么定义"成功放置"?毫米级阈值会被 reward hacking
  4. **失败代价大**:撞断硬件,修复成本高
- **DAgger 范式与 IL 训练对齐**:Groot 不是 RL 训的,DAgger 工程改动小,数据安全可控,**不需要 reward 设计**

### Q11.5 "拟合悖论"(钩子完整答)

**核心答案(讲拟合悖论时用)**:

> "业界要求 95+ 成功率,但**正常的数据规模训不出 95+**。实际上是'**过拟合到某种程度才能达到 95+**'。
>
> DAgger 正好提供这个'针对性过拟合'的路径 —— 但**问题是 DAgger 干预数据的数量永远不够**。"

**为什么数量不够**(被追问时讲):
- **失败模式是 long tail**:估计 100+ 个失败模式
- 每个失败模式至少需要 5-10 条干预数据才能学稳 → **总共 2500-5000 条**
- 每条干预 30 秒人工接管 + 标注 → **5000 条 = 41+ 小时纯人工**,1-2 周成本
- 真实世界的失败模式**远不止 100 个**,DAgger 永远补不完 → 这是悖论

### Q11.6 "拟合悖论怎么解?"(钩子的二级追问)

**四个方向**(我有想过但离职前没验证):

1. **干预数据 weight decay**:刚加进训练集时权重高,后面慢慢降权,避免长期过拟合到 corner case
2. **Curriculum learning**:先训简单 corner case,再训困难的,而不是混在一起
3. **评测分两套**:OOD 和 in-dist 各一套,防止模型修复 OOD 时回退 in-dist 性能
4. **失败模式聚类**:100 个失败模式不一定是 100 类,用模型 hidden state embedding + k-means 聚类后可能只有 10 大类,**每类多采几条更高效**

### Q11.7 "PPO / RL 1 页科普"(对照 DAgger 的弹药)

**RL 基本设定**:agent + env + state + action + reward,目标是累计 reward 最大化。

**On-policy(PPO 是代表)vs Off-policy(SAC, Q-learning)**:

| | On-policy(PPO) | Off-policy(SAC) |
|---|---|---|
| 数据来源 | **当前 policy 自己跑出的** | 任何 policy 的历史数据 |
| Sample Efficiency | **低**(每次更新旧数据废掉) | 高(数据可一直用) |
| 业界用途 | 仿真 RL 主流 | 真机 RL 主流 |

**真机 PPO 的根本问题**:**真机数据贵,PPO 浪费不起**。

**Reward Hacking**(机器人场景例子):
- reward = "箱子被举起来的高度" → 模型可能学会**把箱子撞到桌边卡住假装在举**
- reward = "成功放置(<5cm)" → 模型可能学会**把物体扔到大概位置**而不是稳定放置
- 真机 reward 比仿真难设计:传感器有噪声 / 任务定义模糊 / 安全约束多

**真机 RL 不现实的工程对比**:

| 维度 | 仿真 RL | 真机 RL |
|---|---|---|
| 1 step 时间 | 毫秒级 | 秒级 |
| 数据成本 | 几乎为 0 | 1 episode 30s+ |
| 失败代价 | reset | 撞坏硬件 |
| Sample 数量 | 百万级 | 千级很难 |

**业界主流做法(包括 π0.6)**:
- **不在真机 RL** — 仿真 / 半真实环境
- **IL pretrain + RL fine-tune** — RL 只做最后一公里
- **Off-policy + 离线 RL** — 复用历史数据

**所以"我了解 RL 但选 DAgger"的合理理由**:
- 我们是真机 + IL 训练的 Groot
- RL fine-tune 需要稠密 reward,拆垛任务难设计
- DAgger 与 IL 训练范式对齐,工程改动小,数据安全可控

### Q11.8 "DAgger 的局限?"

- **依赖人工干预成本**:每个 intervention 需要人在场,不能完全自动化
- **专家标注质量**:遥操质量影响上限
- **OOD 边界难定义**:什么算"值得干预",需要人工判断
- 长期替代:**用大 VLA 的 zero-shot 能力做 corner case 处理**(但目前还不够成熟)
