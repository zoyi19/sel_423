# 02 简历条目 —— 中英双语,3 个长度版本(修订版)

> **用途**:直接粘贴到简历上。三个版本按简历空间选用:
> - **3 行版**:简历已经很满,给这个项目留的空间有限
> - **5 行版**(推荐):大多数实习生简历的标准长度
> - **8 行版**:项目是简历绝对主角,占半页
>
> **撰写原则**:
> 1. **动词优先**(主导 / 负责 / 参与 / 完成 / 搭建 / 对比 / 优化)
> 2. **数字优先**(6 个月 / 85% / 30Hz / 2900+ / **30%(UMI 负面结论)**)
> 3. **模糊责任用 "团队"**,个人贡献用 "主导 / 参与"
> 4. **不堆砌名词**,每个技术词要么能解释、要么直接删
> 5. **不出现"我"字**(简历第一人称省略是惯例)
> 6. **UMI + Lingbot-VLA 30% 负面结论必须出现** —— 这是你最独特的亮点

## 词条使用建议

### 英文简历(海外 / 外企)
- 用英文 Version B
- 技术名词不翻译(GR00T / OpenVLA / UMI / Lingbot-VLA / SAM3 保持英文)
- "Owned" 比 "Led" 在 intern 语境下更自然

### 中文简历(国内大厂 / 机器人公司)
- 用中文 Version B
- 技术名词可适度英文(GR00T 1.5 / UMI / SAM3+IC-light)
- "主导 / 参与 / 团队成果" 三类标签严格用,**面试时会反复核验**

### 空间紧张(简历压 1 页)
- 用 Version A
- 保留 3 个核心数字:**6 个月 / 85% / 30%(Lingbot 负面)**
- 可省略 SAM3+IC-light 细节,保留 UMI 负面结论这个亮点

### 简历以项目为主角(研究向 / 博士申请)
- 用 Version C
- 加一段"Key Insights / Findings"讲**个人反思** —— 机器人公司 TL 级别面试官会欣赏

## 面试官对简历的"挑错"预判

1. **"你说主导 UMI,做到哪一步?"**
   → "**完整的预研闭环** —— 方案设计、硬件搭建、软件 pipeline、1000 条数据采集、Lingbot-VLA 训练、量化评测得到 30%。**负面结论也是完整产出**,不是半途而废。"

2. **"85% 是你做的吗?"**
   → 见 `03_STAR讲稿.md` 中的诚实话术。归团队,讲清自己参与点。

3. **"Lingbot-VLA 30% 是不是你没做好?"**
   → "这是一个受控实验得出的结论。数据量、模型选型、任务难度都是合理的。**我的结论是'UMI 在精细操作上不行',不是'UMI 完全不行'**,边界很清楚。"

4. **"为什么用 SAM3 不用 SAM / SAM2?"**
   → "SAM3 在开放词汇和视频场景支持更好,我们的真机采集数据是视频序列,SAM3 更自然。**具体对比 benchmark 我没跑过,选型依据主要是文档和开源社区反馈**。"

5. **"LeRobot v3.0 有什么改进?"**
   → "最大改进是 **Multi-LerobotDataset 并行加载**,其他 API 变化相对小。不熟的就只讲这一点。"

6. **"你主导的三条子线,哪一条最值得讲?"**
   → "**UMI 预研** —— 因为它体现了我在方向判断上的能力,而不只是执行层面。得出 30% 的负面结论比做出一个 60% 的正面结果更能展现研究素养,也对公司决策更有帮助。"

7. **"实习 6 个月主导三条子线,会不会太浮?"**
   → "每条子线时间和边界是清晰的,不是同时跑三条。阶段 1 纯工程化(1 个月),阶段 2 的 DAgger + SAM3 是协同的(3 个月合起来),阶段 3 UMI 是 2 个月的独立预研。**三条子线在时间上是串行的,深度上是递进的**(从工程 → 失败处理 → 预研)。"

8. **"阶段 1 一个月能搭完采数工具链 + pytrees 框架?"**
   → "不是从零搭,是**在现有采数脚本和 py_trees 行为树基础上做工程化和统一接口**。原有的功能继续可用,我的工作是在上面做抽象 + 扩展 + 文档化,这是合理的 1 个月工作量。"
