## Q33 描述智元 LAPA 工作 🟡

**框架**:**LAPA = Latent Action Pretraining from Videos**(智元 + KAIST,NeurIPS 2024)。核心:**从无标签视频里学动作**,不需要 action 标签。

**详解**:
- 问题:VLA 训练需要 (image, action) 配对数据,但**互联网视频没有 action 标签**
- LAPA 解法:
  1. **学一个 latent action quantizer**(VQ-VAE 风格):从 (frame_t, frame_{t+1}) 抽出隐变量 z 当"latent action"
  2. **预训练 VLA**:在大规模视频上学 (image_t, language) → z(latent action)
  3. **下游微调**:用真机数据微调 z → 真实 action 的解码器
- 优势:能利用互联网视频规模(几个数量级大于真机数据)
- **缺点**:latent action 学得不好,精度有损;需要下游做适配

**钩子**(诚实):"LAPA 本身我没复现过,看过论文。这条思路和 GR00T 的 **FLARE loss**(Future Latent Representation Alignment)异曲同工 —— 都是想从无 action 标签的视频里挖信号。"

## Q34 ACT 训练和推理流程 🟢

**框架**:ACT = Action Chunking Transformer(Aloha 论文),三件套:**Action Chunking + Temporal Ensemble + CVAE**。

**训练**:
- 输入:multi-camera images(单帧)+ state + language
- 编码:ResNet/DINO 编码 image,joint pos 投影成 token
- CVAE encoder:`(images, actions) → latent z`(训练时存在,推理时取先验)
- CVAE decoder:`(images, language, z) → action chunk shape (H, action_dim)`,默认 H=50 或 100
- Loss:**KL(z || N(0,I)) + L1(action_pred, action_gt)**

**推理**:
- 从先验 z ~ N(0, I) 采样
- decoder 一次性输出 H 步 action chunk
- 关键 trick:**Temporal Ensemble**
  - 每个时刻 t,有过去多个 chunk 都在预测 t 的 action(因为 chunk 重叠)
  - **加权平均**(越新的 chunk 权重越高)→ 平滑

**钩子**:"我实习项目里,ACT 单摞搬运做到 80% 成功率(团队入职前达成,我后面 pytrees 封装对接)。**ACT 的 chunk size 50 = 0.5 秒**,刚好覆盖一个完整动作单位(伸手→抓→收)。"

## Q37 VLM 相关:Transformer / ViT / 自回归生成全流程 / 模型多少 B 🟡

**框架**:讲三件事 —— Transformer 基础 / ViT 把图分 patch 当 token / 自回归生成是逐 token 用之前的输出做条件。

**详解**:
- **Transformer**:Self-Attention + FFN + LayerNorm,核心是 attention 让 token 互相看,公式 `softmax(QK^T/√d)V`
- **ViT**:把 224×224 图分成 16×16 patch → 196 个 patch token + 1 个 CLS token,接标准 Transformer encoder
- **自回归生成流程**:
  1. tokenize input → 走 transformer 得 hidden state
  2. 取最后 token 的 logits,softmax 出概率分布
  3. 采样下一个 token(greedy / top-k / top-p)
  4. 把新 token 拼到序列末尾,**用 KV cache 跳过前面 token 的 K/V 重算**
  5. 重复直到 EOS 或 max_len

**模型尺寸**(背):
- Gemma-2B(PaliGemma 用):**2B 参数**
- Eagle-2.5(GR00T 用):**~3B**
- LLaMA-2-7B(OpenVLA 用):**7B**

**钩子**:"我实习对接过 PaliGemma-3B(π0.5)和 Eagle-2.5(GR00T 1.5),**两者尺寸相近但前者是 Google 通用 VLM,后者是 NVIDIA 针对机器人调过的**。"

# §三、架构通用:KV / Transformer / Mamba / GQA(8 题)

## Q19a-d KV Cache 一组(缓存什么 / 影响 / GQA / 显存)🟡

### Q19a:KV cache 缓存的是什么?

**框架**:每层 attention 的 **K(Key)和 V(Value)矩阵**,**不缓存 Q**。

**详解**:
- attention 计算 `softmax(QK^T/√d)V`
- K, V 只依赖输入 X 和权重 W_K, W_V → **X 不变,K/V 不变** → 可缓存
- Q 每步都要重新算(因为是"我想问什么")
- 缓存粒度:**每层每 head 一个 (B, N, D_head) 的 K 和 V**

### Q19b:启用 KV cache 对时序动作推理的影响

**正面**:推理速度大幅提升(参考 π0.5 prefix KV cache,~10× 加速)
**负面**:
- **显存占用增加**(下面讲)
- **chunk-level 推理(flow matching)和自回归 LLM 的 cache 形态不同**:
  - LLM:动态增长(每生成 1 token 加 1)
  - flow matching:静态(prefix 一次缓存,suffix 每步 ODE 不进 cache,因为 suffix 内容会变)

### Q19c:GQA / MHA 下 KV cache 怎么变?

**关键差异**:**MHA 每个 head 都有独立 K/V,GQA 多个 head 共享一组 K/V**。

| 配置 | 头数 | K/V 头数 | KV cache 大小 |
|---|---|---|---|
| **MHA** | H | **H**(每个 head 独立) | `B × N × H × D` |
| **GQA** | H | **G < H**(分 G 组) | `B × N × G × D`(**省 H/G 倍**) |
| **MQA** | H | **1**(所有 head 共享) | `B × N × 1 × D`(**省 H 倍**) |

**索引方式**:
- MHA 推理时直接索引 `cache[layer][head]`
- GQA 推理时**多个 head 共用同一份 cache**(repeat / broadcast 即可)

### Q19d:长任务 KV cache 显存溢出怎么办?

**5 个常见手段**:
1. **GQA / MQA** 替换 MHA(直接省 H/G 倍)
2. **PageAttention**(vLLM 提出):分页式管理 cache,避免 fragmentation
3. **Sliding Window**:只保留最近 N 个 token 的 cache(Mistral 用)
4. **量化**(FP8/INT8):cache 用低精度存,省 2-4×
5. **抛弃 cache** 改用 **Mamba / RWKV** 这种 O(N) 内存的架构

**钩子**:"在我做的拆垛任务里,chunk 推理是 flow matching 而非自回归,所以 KV cache 是静态的(prefix 一次缓存),**显存压力小于 LLM 长上下文场景**。但如果做长视地任务(几十秒以上)累积 chunk,这才会变成问题。"

## Q19f Self-Attention 完整计算流程 🟡

**框架**:输入 → QKV 投影 → 注意力分数 → softmax → 加权求和。

**详解**(数学完整版):
```
输入: X ∈ R^{N×D}  (N 个 token,每个 D 维)

1. QKV 投影:
   Q = X·W_Q,  W_Q ∈ R^{D×D_head}
   K = X·W_K,  W_K ∈ R^{D×D_head}
   V = X·W_V,  W_V ∈ R^{D×D_v}

2. 注意力分数(query-key 相似度):
   scores = Q·K^T  ∈ R^{N×N}    # 每个 query 和每个 key 算点积

3. 缩放:
   scores = scores / √D_head     # 防止 softmax 饱和(见下题)

4. Mask(可选):
   scores[mask == 0] = -inf       # 因果 mask / padding mask

5. Softmax:
   attn = softmax(scores, dim=-1) ∈ R^{N×N}

6. 加权求和:
   output = attn · V  ∈ R^{N×D_v}

7. Multi-Head:把上面在 H 个 head 上并行,concat 后线性投影
```

**钩子**:"我读过 openpi 的 `gemma.py` 里 attention 的实现,他们用 einsum 优化:`jnp.einsum('BTKGH,BSKH->BKGTS', q, k)`,**G 是 GQA 的 group 维度**,这样能在一行代码里覆盖 MHA / GQA / MQA 三种情况。"

## Q19g 为什么除以 √d_k?🟡

**框架**:**防止 softmax 进入饱和区**,梯度消失。

**详解**(数学直观):
- Q, K 元素假设独立,均值 0,方差 1
- 点积 `q·k = Σ q_i × k_i`,**方差是 d**(d 个独立项相加)
- d 大时(D_head=128 / 256),方差大 → 点积值有的 +30 有的 -30
- 进 softmax 后:
  ```
  softmax([30, 0, -30]) ≈ [1, 0, 0]   # 一个数主导,梯度集中,其他 0 梯度
  ```
- 除以 √d 把方差归一化到 1,softmax 输出**不会过于尖锐**,梯度均匀分布

**面试可加分句**:"这是 Vaswani 2017 原论文的 Section 3.2.1 给出的具体推导,核心是**让 softmax 输入的方差不随 d_k 增长**。"

## Q19h GQA 结构和作用 🟡

**框架**:**Grouped Query Attention**,Q 分多 head,K/V 分少 group,**多个 Q head 共享一组 K/V**。

**详解**:
- MHA(标准):H 个 head,每个 head 独立 Q/K/V → KV cache 大
- MQA(极端):H 个 Q,**1 个共享 K/V** → KV cache 最小,但效果损失明显
- GQA(中庸):H 个 Q,**G 个 K/V**(G < H,通常 G = H/8) → 接近 MHA 效果,显存接近 MQA

**作用**:
1. **显存节省**:KV cache 大小 ∝ G/H
2. **推理加速**:K/V 计算量小
3. **效果接近 MHA**(LLaMA-2 的实测)

**实例**:LLaMA-2-70B 用 GQA-8(8 组 K/V,64 头 Q),推理显存比 MHA 省 8×。

**钩子**:"看 openpi `gemma.py` 配置,Gemma-2B 用 `num_heads=8, num_kv_heads=1` —— **这是 MQA**(KV 只 1 头),不是 GQA。Google 在 Gemma 上比 LLaMA 更激进。"

## Q19o 推理时机械臂停滞 / 动作不连贯的原因 🟡

**框架**:三大类原因 —— **chunk 边界跳变 / 推理频率不够 / OOD distribution shift**。

**详解**:
1. **Chunk 边界跳变**:相邻 chunk 衔接处动作突变 → 抖动 / 停滞
   - 解法:**Temporal Ensemble + chunk inpainting + 低通滤波**(推理平滑三件套)
2. **推理频率不够**:控制 100Hz,但推理 5Hz → 动作"老旧",环境变化没跟上
   - 解法:**减小 chunk size 或加快 ODE 步数**
3. **OOD distribution shift**:训练没见过的状态 → 速度场预测不准 → 输出"原地踏步"
   - 解法:DAgger 补采 + 数据增强
4. **Action space 设计错误**:cmd_pose 时间不连续 / delta joint pos 累积漂移
   - 解法:**走路用 cmd_vel,手臂用 cmd_pose**

**钩子**:"我实习的拆垛项目踩过这个坑 —— 早期 ACT 推理 chunk 边界明显跳变,**杨小康主导的'推理平滑三件套'**(inpainting + 低通滤波 + 插值)解决了,我后来在 pytrees 推理节点里把这套接入了。"

# §四、Diffusion + Flow Matching + Action Chunk(5 题)

## Q4 VLM + Diffusion 解耦?手术精度场景下的瓶颈?🔴 → 你的版本

**🔴 原题**(手术):大模型直接输出坐标点,在手术精度场景下最大瓶颈?

**通用回答**:
- **解耦 VLM 和 Diffusion 的理由**:
  - VLM 擅长 high-level 语义(看到 / 理解 / 规划),**不擅长精细动作**(连续动作的细微变化)
  - Diffusion / Flow Matching 擅长**学复杂分布**(多峰 / 连续),适合 low-level action
  - 解耦让两者各司其职 → 训练更稳,精度更高
- **大模型直出坐标点的瓶颈**(精度场景):
  1. **token 离散化精度损失**(把毫米级位姿离散成 1024 个 bin,精度崩)
  2. **多峰分布表达力不足**(softmax 出 single-peak,处理不了"两个合理路径")
  3. **梯度信号弱**(回归任务用交叉熵 loss 不友好)

**🟢 你的对应版本**:虽然你没做手术,但你做拆垛**也是精度敏感场景**。
- 你看过 GR00T N1.5 用 **DiT + Flow Matching action head**(冻结 VLM)而不是端到端
- 这就是工业 VLA 圈的共识:**VLM 当感知,小模型当 action 头**
- 实习里 4 类箱体混训 85%,**如果改成端到端直出 token,大概率 OOD 倾斜从 0% 直接变 0% 全任务**

**钩子**:"如果让我类比手术,**精细操作的本质问题是 embodiment gap 的放大** —— 这正是我 UMI 实验得出 30% 负面结论的根源(无本体相机视角和机器人视角差,精细任务被放大)。"

## Q5 Action Chunk 闭环修正(手术组织形变实时变)🔴 → 你的版本

**🔴 原题**:chunk 没跑完,组织已经变了,怎么闭环?

**通用回答**:**Receding Horizon + Temporal Ensemble**,本质是"高频重推理 + 多 chunk 加权融合"。

具体做法:
- **不等 chunk 跑完就重新推理**(每 5-10 步重推一次)
- 每个时刻 t 有多个旧 chunk 和新 chunk 都在预测 t 的动作
- **加权平均**(越新权重越高)→ 自动适应环境变化
- 这是 ACT 论文的 **Temporal Ensemble** 核心

**🟢 你的对应版本**:拆垛任务里也有这个问题。
- GR00T 输出 16 步 chunk,执行第 5 步时箱子可能被外力撞了
- 我们的处理:**每 5 步重新推理 + temporal ensemble**
- 实习时杨小康主导的"推理平滑三件套"(inpainting + 低通滤波 + 插值)就是工程化版本

**钩子**:"流式推理是 VLA 通用挑战。手术比拆垛更极端的是**毫米级精度 + 实时形变**,**chunk size 可能要从 16 降到 4-8**(GR00T 是 16,π0.5 是 50,手术大概率 < 16)。"

## Q6 Diffusion 多峰分布(手术避障两条路径)🔴 → 你的版本

**🔴 原题**:两条都合理的避障路径,Diffusion 怎么避免决策摇摆?

**通用回答**:**Diffusion 天然支持多峰**(这是它比 MSE-based BC 强的核心),但**摇摆**问题靠**chunk 一致性 + temporal ensemble**解决。

为什么 Diffusion 支持多峰:
- MSE BC:`L = (a_pred - a_gt)²`,**强迫输出位于多峰的"中间"**(两条路径的中点 → 撞墙)
- Diffusion / Flow Matching:**学整个 action 分布**,推理时**采样某一个 mode**(不是平均)

**为什么不会摇摆**:
- 一次推理出 chunk(H 步)→ chunk 内部一致(都 commit 到一条路径)
- temporal ensemble 加权时旧 chunk 主导 → **不会逐 step 切换 mode**

**🟢 你的对应版本**:拆垛里"先抓哪个箱"也是多峰。
- 实习里 Groot N1.5 看到两个相同箱子时不会摇摆,**因为 chunk 一次性 commit 16 步动作**
- 推理时如果突然换 mode → "推理平滑三件套"会平滑掉

**钩子**:"多峰是 Diffusion / Flow Matching 比 ACT(MSE-based BC)的关键优势。我实习对照过 ACT 和 GR00T 在'两个箱子相同'场景的输出,**ACT 偶尔会输出'平均位置'(撞中间)**,GR00T 不会。"

## Q42 / Q47 / Q48 Diffusion Policy 数据流 / RL 微调对比 🟡

**Q46 Diffusion Policy 数据流**:

```
1. 数据采集:
   (image, state, action_chunk) — 真机遥操或仿真

2. 训练:
   - input: image_t (CNN encoder) + state_t
   - condition: cond = encoder(image, state)
   - DDPM noise schedule: a_t = α_t·a_clean + σ_t·ε
   - 模型 ε_θ(a_t, t, cond) 预测噪声 ε
   - Loss: MSE(ε_θ(a_t, t, cond), ε)

3. 推理:
   - a_T ~ N(0, I)
   - for t = T, T-1, ..., 1:
       ε_pred = ε_θ(a_t, t, cond)
       a_{t-1} = denoise step
   - 输出 a_0 = action chunk
   - 通常 T = 100,DDIM 加速到 10-20 步
```

**Q47 为什么用 Diffusion Policy 框架,试过 RL 吗?**:
- IL(Diffusion Policy):**学专家分布**,真机数据贵但学到的策略稳
- RL:**奖励驱动**,真机不现实(见 Q44)
- **试过 RL 但回到 IL** 是工业界主流路径(包括 PI 的 π*0.6,也是 IL pretrain → RL fine-tune,不是从零 RL)

**Q48 vs RL 微调方案优势**:
- **样本效率**:Diffusion Policy 几百 demos 可用,RL 要 10^6+ 步
- **安全**:不会撞坏机器
- **可解释**:直接学专家行为,不会 reward hacking

**钩子**:"我实习项目里用过 GR00T 的 Flow Matching head(也是 Diffusion 系),**没用 RL 因为我们的 DAgger 范式更对齐 IL 训练**。π*0.6 的 RECAP 是 RL 后训练的代表,但它是建立在已有 IL 基础上的精修,不是从零 RL。"

# §七、★ RL 详解(12 题,深度 C 版)

## Q39 PPO 中的 log p 是怎么算的?⭐

**框架**:`log π(a|s)` 是策略输出动作 a 在状态 s 下的对数概率。**离散用 log_softmax**,**连续用 Gaussian log_prob**。

**详解**:

### 离散动作(DQN / 离散 PPO)

```python
# 策略输出 logits ∈ R^A(A 个动作类别)
logits = policy(state)                          # (B, A)
log_probs = F.log_softmax(logits, dim=-1)        # (B, A)
log_p_action = log_probs.gather(1, action.unsqueeze(1))  # (B, 1) 取所选动作的 log_p
```

### 连续动作(机器人 PPO 主流)

策略输出**高斯分布的均值 μ 和 log_std**(log std 而非 std,保数值稳定)。

```python
# Policy: state → (μ, log_std)
mu = policy_mean(state)                 # (B, action_dim)
log_std = policy_logstd(state)          # (B, action_dim) 或可学习的全局参数
std = log_std.exp()

# action 通常 reparametrize 采样
eps = torch.randn_like(mu)
action = mu + std * eps                 # 重参数化,梯度可流

# log_prob(diagonal Gaussian)
def gaussian_log_prob(action, mu, log_std):
    """对每一维独立高斯求 log p,然后 sum"""
    var = (2 * log_std).exp()
    log_p = -0.5 * ((action - mu) ** 2) / var - log_std - 0.5 * math.log(2 * math.pi)
    return log_p.sum(dim=-1)            # (B,) per-step log probability
```

数学公式:
```
log N(a; μ, σ²) = -0.5·((a - μ)/σ)² - log σ - 0.5·log(2π)
```

### 在 PPO 里怎么用

```python
log_p_old = collect_data_with_old_policy(...)   # 数据收集时存
# 训练时
mu_new, logstd_new = policy(states)
log_p_new = gaussian_log_prob(actions, mu_new, logstd_new)
ratio = (log_p_new - log_p_old).exp()           # importance ratio
# clip surrogate loss
L_clip = -torch.min(ratio * advantage,
                     torch.clamp(ratio, 1 - eps, 1 + eps) * advantage).mean()
```

**钩子**(诚实):"我在实习里没直接训过 PPO,但读过 SimpleVLA-RL 和 RECAP 论文,**他们用的是 GRPO 而不是 PPO**(advantage-conditioned policy)—— 工程上更稳,不需要严格的 importance sampling。"

**RL 算法五兄弟**:

| 算法 | On/Off-policy | Action 类型 | 关键 trick |
|---|---|---|---|
| **DQN** | Off | 离散 | Replay buffer + Target network |
| **DDPG** | Off | 连续 | Actor-Critic + Target network |
| **TD3** | Off | 连续 | DDPG 改进:**Double Q + Delayed update + Target policy smoothing** |
| **PPO** | On | 离散/连续 | Clip ratio + GAE + KL constraint |
| **SAC** | Off | 连续 | **Max Entropy + Soft Q + 自动温度** |

### SAC 详解(连续动作 RL 主流)⭐

**核心思想**:**最大化 reward 的同时最大化 policy 的熵**(鼓励探索),目标:

```
J(π) = E[Σ_t (r_t + α · H(π(·|s_t)))]
```

其中 H 是策略熵,α 是温度参数(控制 reward vs 熵的 tradeoff)。

**三个网络**:
1. **Actor π(a|s)**:输出 Gaussian (μ, σ)
2. **Two Q networks Q1, Q2**(Double Q,防 overestimation)
3. **Target Q networks**(EMA copy of Q1, Q2)

**Critic 更新**(Soft Bellman):
```python
# Soft target
with torch.no_grad():
    next_a, next_log_p = actor.sample(next_state)              # reparametrize
    target_q1 = target_q1_net(next_state, next_a)
    target_q2 = target_q2_net(next_state, next_a)
    target_q_min = torch.min(target_q1, target_q2)             # 取小,防 overestimation
    soft_target = target_q_min - alpha * next_log_p           # 减熵 → soft value
    target = reward + gamma * (1 - done) * soft_target

q1_loss = F.mse_loss(q1_net(s, a), target)
q2_loss = F.mse_loss(q2_net(s, a), target)
```

**Actor 更新**(最大化 Q - α·log_π):
```python
new_a, log_p = actor.sample(state)              # reparam
q1 = q1_net(state, new_a)
q2 = q2_net(state, new_a)
q_min = torch.min(q1, q2)
actor_loss = (alpha * log_p - q_min).mean()    # 最小化 = 最大化 Q - α·log_p
```

**温度自动调整**(SAC v2):
```python
target_entropy = -action_dim                    # heuristic
alpha_loss = -(log_alpha * (log_p + target_entropy).detach()).mean()
```

**为什么 SAC 比 PPO 更适合真机**:
- **Off-policy** → 复用历史数据,样本效率高(真机数据贵不能浪费)
- **Max entropy** → 探索充分,不容易卡局部最优
- **连续动作天然支持**(机器人 action 是连续的)

**钩子**:"如果让我做真机 RL,**我大概率选 SAC + DAgger 风格的人工干预**(类似 HIL-SERL),不会选 PPO —— PPO 的 on-policy 性质让真机数据用一次就废,工程不现实。"

## Q41 online / offline 数据池调用,采到 online 失败数据怎么办 ⭐

**框架**:三种处理 —— **加权降权 / 失败专属 buffer / 提取教训样本**。

**详解**:

### 数据池设计(常见做法)

```python
class HybridReplayBuffer:
    def __init__(self):
        self.online_buffer = deque(maxlen=100000)        # 在线 RL 收集的
        self.offline_buffer = deque(maxlen=1000000)      # 离线专家数据
        self.intervention_buffer = deque(maxlen=50000)   # 人工干预专属

    def sample(self, batch_size):
        # 按比例混合采样
        n_online = int(batch_size * 0.5)
        n_offline = int(batch_size * 0.3)
        n_intervention = batch_size - n_online - n_offline
        return (sample(self.online_buffer, n_online) +
                sample(self.offline_buffer, n_offline) +
                sample(self.intervention_buffer, n_intervention))
```

### 失败数据处理三策略

**策略 1:加权降权(treat as bad reward)**
- 失败 episode 的 reward = 0(或负值)
- 模型自然学到"避开这种状态"
- 适合 sparse reward 任务

**策略 2:失败专属 buffer + DAgger 干预**
- 失败 episode 触发人工干预
- 干预数据进 `intervention_buffer`(权重高)
- 训练时混合采样,**让模型学"failure 状态下专家怎么处理"**
- 这就是 HIL-SERL / RECAP 的核心思想

**策略 3:提取"教训样本"**
- 失败前几步可能是关键决策点
- 标注这些步骤为 "high-leverage state"
- 单独 oversample 这些状态训练

**钩子**:"**这正好是我实习里 DAgger 失败恢复做的事** —— 失败 trajectory 通过 VR/键盘接管,干预数据混合缓冲 10-30% 加入下一轮训练。在 IL 里叫 DAgger,在 RL 里叫 HIL-SERL,**底层逻辑一致**。"

## Q42(下) PPO 对动作快(action chunk)是怎么处理的?⭐

**框架**:**Chunk-level GAE / 把 chunk 当 macro-action / 用 Diffusion-PPO**。

**详解**:

### 问题来源

PPO 经典版本是 **per-step**(每步 ratio + advantage),但 VLA 输出 chunk(H 步连续动作),怎么搞?

### 三种主流处理

**方法 1:Chunk-level GAE**
- 把整个 chunk 当一个"宏观动作"
- log_p(chunk) = Σ log_p(a_i)(独立性假设)
- advantage 用 chunk-level reward 累计
- ratio = exp(log_p_new(chunk) - log_p_old(chunk))

**方法 2:Per-step PPO + chunk policy**
- chunk 内每步当独立 action
- 但 policy 是 chunk-level 的(一次 forward 出 H 步)
- 训练时 unroll chunk,对每步算 ratio
- 实现复杂,但精细度高

**方法 3:Diffusion-PPO(DPPO,Princeton 2024)**
- 把 Diffusion 的 denoising 步骤当一个"trajectory"
- 在 denoising trajectory 上跑 PPO
- 每个 denoising step 有自己的 ratio
- 适合 Diffusion Policy 的 RL 微调

### 工程现实

社区共识:**chunk + RL 没有统一最佳方案**,GRPO / RECAP 这种 advantage-conditioned 方法(不是经典 PPO)反而**绕开了这个问题** —— 不需要 importance sampling,直接 condition on advantage。

**钩子**:"**这也是为什么 π*0.6 用 RECAP 而不是 PPO** —— PPO + chunk 工程上很难调,advantage-conditioned policy 简单粗暴反而管用。我读 RECAP 论文时印象很深。"

## Q43 手撕 diffusion / PPO 代码 ⭐

### Diffusion 训练 loop(30 行核心)

```python
class DiffusionPolicy(nn.Module):
    def __init__(self, action_dim, horizon, T=100):
        super().__init__()
        self.T = T
        self.noise_pred = UNet(action_dim, horizon)            # 预测噪声
        # cosine noise schedule
        self.register_buffer('alphas', cosine_schedule(T))     # ᾱ_t

    def train_step(self, action_clean, condition):
        B = action_clean.shape[0]
        t = torch.randint(0, self.T, (B,), device=action_clean.device)
        noise = torch.randn_like(action_clean)

        sqrt_alpha_bar = self.alphas[t].sqrt().view(-1, 1, 1)
        sqrt_1m_alpha_bar = (1 - self.alphas[t]).sqrt().view(-1, 1, 1)
        action_t = sqrt_alpha_bar * action_clean + sqrt_1m_alpha_bar * noise

        noise_pred = self.noise_pred(action_t, t, condition)
        loss = F.mse_loss(noise_pred, noise)
        return loss

    @torch.no_grad()
    def sample(self, condition, num_steps=10):  # DDIM 加速
        a = torch.randn(...)                                    # a_T
        for t in reversed(torch.linspace(0, self.T-1, num_steps).long()):
            noise_pred = self.noise_pred(a, t, condition)
            a = ddim_step(a, noise_pred, t)                     # 数学略
        return a
```

### PPO clip surrogate loss(20 行核心)

```python
def ppo_loss(states, actions, advantages, returns, log_probs_old,
             policy, value_net, eps=0.2):
    # === Actor loss ===
    mu, log_std = policy(states)
    log_probs_new = gaussian_log_prob(actions, mu, log_std)
    ratio = (log_probs_new - log_probs_old).exp()
    surrogate1 = ratio * advantages
    surrogate2 = torch.clamp(ratio, 1 - eps, 1 + eps) * advantages
    actor_loss = -torch.min(surrogate1, surrogate2).mean()

    # === Critic loss ===
    values = value_net(states)
    critic_loss = F.mse_loss(values, returns)

    # === Entropy bonus(鼓励探索)===
    entropy = 0.5 * (1 + (2 * log_std + math.log(2 * math.pi))).sum(-1).mean()

    total = actor_loss + 0.5 * critic_loss - 0.01 * entropy
    return total
```

### GAE(advantage 估计)

```python
def compute_gae(rewards, values, gamma=0.99, lam=0.95):
    advantages = torch.zeros_like(rewards)
    gae = 0
    for t in reversed(range(len(rewards))):
        delta = rewards[t] + gamma * values[t+1] - values[t]
        gae = delta + gamma * lam * gae
        advantages[t] = gae
    returns = advantages + values[:-1]
    return advantages, returns
```

## Q35 RTC 实现 🟡

**框架**:**RTC = Real-Time Chunking**(Receding Trajectory Control 的 VLA 变体),sliding window 推理 + chunk overlap。

**详解**:
- 不等当前 chunk 跑完,**每 N 步就重新推理新 chunk**
- 多个旧/新 chunk 在每个时刻都给出动作预测
- **加权平均**(temporal ensemble)→ 平滑

**实现伪代码**:
```python
chunk_buffer = []                    # 存活跃 chunks
for step in range(total):
    if step % refresh_interval == 0:
        new_chunk = policy.sample(observation)
        chunk_buffer.append({'chunk': new_chunk, 'start_step': step})

    # 当前时刻有效 chunks
    active = [c for c in chunk_buffer if c['start_step'] <= step < c['start_step'] + chunk_size]

    # 加权平均(越新权重越高)
    weights = [exp(-(step - c['start_step']) / tau) for c in active]
    action = weighted_avg([c['chunk'][step - c['start_step']] for c in active], weights)
    execute(action)
```

**钩子**:"RTC 不是单独算法,是 ACT 论文里 Temporal Ensemble 的工程实现。**我实习的拆垛任务用了类似机制**,杨小康主导的'推理平滑三件套'就是这个思路。"
